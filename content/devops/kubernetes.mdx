## Зачем это вообще

Представь что у тебя 50 серверов и 30 разных сервисов. Каждый сервис нужно разложить на серверах, следить чтобы не упал, перекатывать без даунтайма, ходить логи смотреть, балансировать трафик. Если делать это руками — ты только этим и будешь заниматься. Kubernetes — это менеджер, которому ты говоришь «вот эти приложения, развёрни их и сделай чтобы всегда работали». Дальше он сам.

## Аналогия

<AnalogyBox>
Представь склад с сотнями работников. **Бригадир** (Deployment) держит наряд: «на упаковке должно быть 5 человек». Кто-то ушёл — бригадир сам найдёт замену. **Диспетчер** (Service) — клиенты звонят на один номер, он сам соединяет со свободным работником. **Охранник на входе** (Ingress) пропускает посетителей в нужный цех по имени и пропуску. А **управляющий складом** (control plane) видит всю картину и решает, кого куда поставить.
</AnalogyBox>

Когда ты делаешь `kubectl apply`, кубернетес сравнивает что ты хочешь (в манифесте) с тем что сейчас есть (в etcd) — и подгоняет реальность под желаемое. Это **reconciliation loop** — главная идея, всё остальное из неё растёт.

## Ключевые концепции

<ConceptGrid>
  <ConceptCard title="Pod" icon="Boxes">
    Минимальная единица планирования. Один или несколько контейнеров с общим сетевым
    пространством и томами. IP у пода эфемерный — умер под, IP пропал. Поэтому Pod
    не используют напрямую, только через Deployment или StatefulSet.
  </ConceptCard>

  <ConceptCard title="Deployment" icon="Layers">
    Управляет ReplicaSet, который держит N копий пода. `kubectl rollout undo` откатывает
    на предыдущий ReplicaSet. Не используй для stateful-сервисов с PVC — там нужен
    StatefulSet, который даёт стабильное имя и порядок запуска/остановки.
  </ConceptCard>

  <ConceptCard title="Service" icon="Network">
    Стабильный виртуальный IP и DNS-имя перед группой подов. Трафик идёт только на поды
    с совпадающими `labels`. `ClusterIP` — внутри кластера, `NodePort` — снаружи через
    порт ноды, `LoadBalancer` — облачный балансировщик.
  </ConceptCard>

  <ConceptCard title="Ingress" icon="Route">
    HTTP(S)-роутер: один внешний IP, маршрутизация по хосту и пути к нужным Service.
    Сам по себе Ingress — только правила; нужен Ingress Controller (nginx, Traefik).
    Без TLS-секрета трафик идёт в открытом виде.
  </ConceptCard>

  <ConceptCard title="ConfigMap и Secret" icon="Key">
    ConfigMap хранит незащищённые конфиги, Secret — то же самое, но base64-кодирован.
    Это **не шифрование** — любой с доступом к кластеру читает Secret через
    `kubectl get secret`. Для настоящих секретов — Vault или External Secrets Operator.
  </ConceptCard>

  <ConceptCard title="HPA" icon="TrendingUp">
    Horizontal Pod Autoscaler масштабирует количество реплик Deployment по метрикам
    (CPU, память, custom). Требует `requests.cpu` в манифесте — без них HPA не знает
    от чего считать процент загрузки и просто ничего не делает.
  </ConceptCard>

  <ConceptCard title="Namespace" icon="Layers">
    Логическая граница: ресурсы разных namespace не видят друг друга по умолчанию,
    плюс на namespace можно навесить квоту (`ResourceQuota`) и сетевую политику
    (`NetworkPolicy`). Прод и stage держи в разных namespace, а ещё лучше — кластерах.
  </ConceptCard>

  <ConceptCard title="PVC / PV" icon="HardDrive">
    PersistentVolume — кусок хранилища (диск, NFS, EBS). PersistentVolumeClaim —
    заявка пода: «мне нужен 10Gi с SSD». Kubernetes сам связывает PVC с PV.
    Без StatefulSet PVC привяжется к **одному** поду, перенос на другую ноду может
    не сработать (depends на storageClass).
  </ConceptCard>
</ConceptGrid>

## Жизненный цикл пода

<MermaidDiagram label="что происходит после kubectl apply" code={`
flowchart TD
  S1[1. kubectl apply<br/>API server валидирует] --> S2
  S2[2. записать в etcd<br/>желаемое: 3 пода] --> S3
  S3[3. deployment-controller<br/>создаёт ReplicaSet → 3 Pod Pending] --> S4
  S4[4. scheduler<br/>выбирает ноду по requests] --> S5
  S5[5. kubelet на ноде<br/>pull image, start container] --> S6
  S6{6. readinessProbe?}
  S6 -- не прошла --> S6r[ждать, пробовать ещё]
  S6r --> S6
  S6 -- прошла --> S7[7. endpoints-controller<br/>добавить Pod IP в Service]
  S7 --> S8([8. Service балансирует на Pod ✓])

  S8 -.-> L1{livenessProbe<br/>периодически}
  L1 -- ок --> L1
  L1 -- провал --> L2[kubelet перезапускает<br/>контейнер<br/><br/>Pod НЕ пересоздаётся,<br/>IP остаётся]
`} />

## Примеры

**deployment.yaml — production-ready nginx**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: production
  labels:
    app: web
    version: "1.27"
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1   # не больше 1 пода недоступно во время обновления
      maxSurge: 1         # временно можно поднять на 1 под сверх replicas
  template:
    metadata:
      labels:
        app: web
        version: "1.27"
    spec:
      # не давать всем подам жить на одной ноде — упадёт нода, всё упадёт
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: DoNotSchedule
        labelSelector:
          matchLabels:
            app: web
      containers:
      - name: nginx
        image: nginx:1.27-alpine
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 100m      # гарантировано зарезервировано на ноде
            memory: 64Mi
          limits:
            cpu: 500m      # выше — throttling, не kill
            memory: 256Mi  # выше — OOMKill
        readinessProbe:
          httpGet:
            path: /healthz
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 10
          # под не получит трафик пока проба не пройдёт
        livenessProbe:
          httpGet:
            path: /healthz
            port: 80
          initialDelaySeconds: 15
          periodSeconds: 20
          # провал — перезапуск контейнера, а не всего пода
```

**service.yaml + ingress.yaml**

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web
  namespace: production
spec:
  selector:
    app: web          # трафик идёт на поды с этим лейблом
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP     # только внутри кластера; Ingress сам выставит наружу
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web
  namespace: production
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - app.example.com
    secretName: app-tls   # kubectl create secret tls app-tls --cert=... --key=...
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web
            port:
              number: 80
```

**configmap + secret + monтирование в Pod**


**Диагностика и управление — kubectl**

```bash
# что вообще происходит в namespace
kubectl get pods -n production -o wide
kubectl get events -n production --sort-by=.lastTimestamp | tail -20

# почему под застрял (Pending / CrashLoopBackOff / ImagePullBackOff)
kubectl describe pod -n production web-7d9f8b-xk2p4

# логи деплоймента, все реплики, follow
kubectl logs -n production -f deploy/web --max-log-requests=5

# статус rolling update — ждёт ли он чего-то
kubectl rollout status deploy/web -n production
kubectl rollout history deploy/web -n production

# откат на предыдущую ревизию
kubectl rollout undo deploy/web -n production

# принудительный перезапуск (создаёт новый ReplicaSet, плавно)
kubectl rollout restart deploy/web -n production
```

## Грабли

<ul className="pitfalls">
<li>**`image: myapp:latest`** — Kubernetes считает образ неизменным, поэтому `rollout` ничего не перекатывает. Тэгай конкретно (`1.4.2`, sha), плюс `imagePullPolicy: Always` если правда хочешь latest.</li>
<li>**`replicas: 1` без `PodDisruptionBudget` и `topologySpreadConstraints`** — нода ушла на drain, под пересоздался на другой ноде с downtime в десятки секунд. Минимум 2 реплики + anti-affinity.</li>
<li>**Слишком агрессивный `livenessProbe`** — приложение долго стартует (миграции БД, прогрев кеша), проба не успевает, под перезапускается, по кругу. Поставь `initialDelaySeconds` с запасом или используй `startupProbe`.</li>
<li>**Нет `resources.requests`** — HPA молчит (нечего считать), scheduler может набить ноду подами и она OOM-нется целиком. Всегда задавай requests, limits — опционально.</li>
<li>**`kubectl delete pod` для рестарта** — Deployment моментально создаст новый, ок, но если делаешь массово — увидишь спайк latency. Правильно: `kubectl rollout restart deploy/...` — плавно, по одному поду.</li>
<li>**`emptyDir` для важных данных** — это диск НА ноде, пропадает вместе с подом. Для базы/файлов используй PVC c подходящим storageClass.</li>
<li>**Secret в git** — даже зашифрованным base64 коммитить нельзя. SealedSecrets, SOPS, Vault, External Secrets Operator — выбирай.</li>
<li>**`kubectl edit deploy ...` на проде** — изменение пропадёт при следующем `kubectl apply` из CI/CD и тебя ждёт ночной звонок. Все изменения через манифест в git.</li>
</ul>

## Когда что выбирать

| Хочу X | Используй |
| --- | --- |
| Stateless сервис (web, api, worker) | `Deployment` |
| База или кластер с порядком и стабильным именем (Postgres, Kafka, Elasticsearch) | `StatefulSet` |
| Запустить задачу один раз до завершения (миграция, бэкап) | `Job` |
| По расписанию — каждый день в 3 ночи | `CronJob` |
| Один экземпляр на каждой ноде (log shipper, node-exporter) | `DaemonSet` |
| Внутренний сервис, не должен торчать наружу | `Service` `ClusterIP` |
| Один внешний IP на много сервисов с HTTP-роутингом | `Ingress` + один `LoadBalancer` |
| Любой L4-трафик наружу (gRPC, базы) | `Service` `LoadBalancer` или `NodePort` |

## Шпаргалка

<div className="cheatsheet">

```bash
# создать / обновить ресурсы из манифеста
kubectl apply -f manifest.yaml
kubectl apply -k ./overlays/prod    # kustomize

# узнать состояние
kubectl get pods,svc,ingress -n prod
kubectl get pod NAME -n prod -o yaml | less
kubectl describe pod NAME -n prod
kubectl logs -f deploy/NAME -n prod
kubectl get events -n prod --sort-by=.lastTimestamp

# раскатка
kubectl rollout status   deploy/NAME -n prod
kubectl rollout history  deploy/NAME -n prod
kubectl rollout undo     deploy/NAME -n prod
kubectl rollout restart  deploy/NAME -n prod

# скейлинг
kubectl scale deploy/NAME --replicas=5 -n prod

# зайти внутрь / port-forward
kubectl exec -it pod/NAME -n prod -- sh
kubectl port-forward svc/NAME 8080:80 -n prod

# секреты и configmap
kubectl create secret    generic NAME --from-literal=KEY=VAL -n prod
kubectl create configmap NAME --from-file=./config.json     -n prod

# контексты (несколько кластеров)
kubectl config get-contexts
kubectl config use-context prod
```
