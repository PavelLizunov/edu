## Зачем это вообще

Без TLS трафик между браузером и сервером идёт в открытом виде — любой роутер между ними читает пароли, токены, тело запросов. В кофейне один `tcpdump` на общем Wi-Fi и вся сессия у злоумышленника. TLS решает три задачи сразу: шифрует трафик, аутентифицирует сервер (ты говоришь с тем кем думаешь, а не с MITM-прокси), и проверяет целостность — данные не были изменены по дороге. Вот почему `http://` на продакшне — это не «надо бы перейти», а «уже горит».

## Аналогия

<AnalogyBox>
Приходишь на пограничный пункт. <b>Пограничник</b> (браузер) не знает тебя лично, но доверяет <b>паспортному столу</b> (Root CA) — его подпись в паспорте легальная. Ты показываешь <b>паспорт</b> (сертификат сервера), пограничник проверяет подпись паспортного стола и убеждается: это реально ты. Дальше вы за секунду договариваетесь на каком <b>шифрованном языке</b> (cipher suite) будете говорить — и дальше переходите на шёпот, который никто вокруг не поймёт.
</AnalogyBox>

Браузер доверяет Root CA из встроенного хранилища (в macOS — Keychain, в Android — свой список, чуть строже). Промежуточный CA живёт онлайн и выдаёт реальные сертификаты — если он скомпрометирован, его отзывают не трогая Root. Сервер должен отдавать leaf + все intermediate в цепочке; Root браузер знает сам.

## Ключевые концепции

<ConceptGrid>
  <ConceptCard title="Симметричное vs асимметричное" icon="KeyRound">
    Асимметричное (RSA, ECDSA) — медленное, используется только в handshake для аутентификации и обмена ключом. Симметричное (AES-256-GCM, ChaCha20) — быстрое, шифрует весь трафик сессии. TLS применяет оба: асимметричное чтобы безопасно договориться о ключе, симметричное — чтобы передавать данные на полной скорости.
  </ConceptCard>

  <ConceptCard title="X.509 — формат сертификата" icon="FileText">
    Сертификат содержит: `subject` (кому выдан), `issuer` (кто подписал), `validity` (notBefore / notAfter), `subjectAltName` (SAN — список доменов и IP), публичный ключ, подпись CA. С 2017 года Chrome не принимает сертификаты без SAN — только `CN` в `subject` недостаточно. Wildcard `*.example.com` покрывает `api.example.com`, но не `a.api.example.com` — только один уровень.
  </ConceptCard>

  <ConceptCard title="Цепочка доверия" icon="Link">
    Root CA → Intermediate CA → leaf. Root хранится офлайн: если его скомпрометировать, рушится доверие ко всем сертификатам под ним. Intermediate CA работает онлайн, выдаёт leaf, его можно отозвать без пересборки Root. Сервер при TLS handshake обязан передать leaf + все intermediate — без этого Android и строгие клиенты вернут `CERTIFICATE_VERIFY_FAILED`.
  </ConceptCard>

  <ConceptCard title="TLS 1.3 Handshake" icon="Shuffle">
    Занимает 1 RTT вместо 2 в TLS 1.2. Client посылает `ClientHello` с cipher list, SNI и сразу `key_share` (эфемерный DH). Сервер отвечает `ServerHello` + сертификат + `Finished` — всё за один round-trip. Данные текут сразу после первого ответа сервера. TLS 1.0 и 1.1 отключены в современных браузерах с 2020 года.
  </ConceptCard>

  <ConceptCard title="SNI — один IP, много доменов" icon="Network">
    Server Name Indication: клиент передаёт hostname в `ClientHello` до того как TLS-соединение установлено. Сервер (nginx) смотрит на него и выбирает нужный `ssl_certificate`. Без SNI на одном IP мог жить только один TLS-сайт. Важно: SNI передаётся в открытом виде — провайдер видит к какому домену ты подключаешься (но не тело запроса).
  </ConceptCard>

  <ConceptCard title="mTLS — клиент тоже предъявляет сертификат" icon="ShieldCheck">
    В обычном TLS только сервер аутентифицируется. mTLS требует сертификат и от клиента. Сервер проверяет его против `ssl_client_certificate` (CA клиентских сертификатов). Стандарт внутри service mesh (Istio, Linkerd) и Zero Trust. Грабля: если у клиента нет сертификата или он подписан другим CA — коннект рвётся с `SSL handshake failed`, а не с понятной ошибкой аутентификации.
  </ConceptCard>

  <ConceptCard title="Let's Encrypt + ACME" icon="Globe">
    ACME-протокол автоматизирует выдачу и обновление сертификатов. `HTTP-01` challenge: LE кладёт токен по `/.well-known/acme-challenge/<token>` и проверяет его через HTTP — порт 80 должен быть открыт. `DNS-01`: создаёт TXT-запись в DNS-зоне — работает за NAT, для wildcard-сертификатов. Срок сертификата LE — 90 дней, обновлять стоит на 60-й день. Certbot делает это автоматически через `systemd timer` или `cron`.
  </ConceptCard>

  <ConceptCard title="Forward Secrecy — компромисс ключа не открывает прошлое" icon="Shield">
    TLS 1.3 использует только ephemeral Diffie-Hellman (ECDHE). Каждая сессия договаривается о своём одноразовом ключе, который нигде не сохраняется. Если через год кто-то получит приватный ключ сервера — записанный трафик прошлых сессий расшифровать всё равно нельзя. TLS 1.2 без DHE (например, `TLS_RSA_*` cipher suites) forward secrecy не даёт — весь трафик расшифровывается одним приватным ключом сервера.
  </ConceptCard>
</ConceptGrid>

## Жизненный цикл TLS 1.3 handshake

<MermaidDiagram label="TLS 1.3 handshake — 1 RTT до данных" code={`
sequenceDiagram
  participant C as Клиент (браузер)
  participant S as Сервер (nginx)

  C->>S: ClientHello<br/>cipher list, SNI: example.com<br/>key_share (ECDHE публичный)
  Note over S: выбирает cipher,<br/>генерирует key_share

  S->>C: ServerHello (выбранный cipher)<br/>+ Certificate (leaf + intermediate)<br/>+ CertificateVerify (подпись)<br/>+ Finished (MAC)

  Note over C: проверяет цепочку<br/>Root CA из trust store<br/>→ OK
  Note over C: вычисляет общий<br/>session key из ECDHE

  C->>S: Finished (MAC)

  C->>S: Application Data (зашифровано)
  S->>C: Application Data (зашифровано)

  Note over C,S: всё это — 1 RTT<br/>после TCP SYN/ACK
`} />

Весь handshake занимает один round-trip поверх TCP. В TLS 1.2 было два RTT: отдельный `ServerHello`, потом отдельный `ChangeCipherSpec`. `0-RTT` в TLS 1.3 (early data) позволяет послать данные вместе с первым пакетом, но не защищён от replay-атак — не используй для не-идемпотентных запросов.

## Примеры

**Дебаг: что сервер реально отдаёт на 443**

```bash
# Показывает полную цепочку, срок и cipher suite.
# -servername обязателен для SNI: без него получишь дефолтный cert хоста.
openssl s_client -connect example.com:443 \
  -servername example.com \
  -showcerts 2>/dev/null \
  | openssl x509 -noout -dates -subject -issuer

# Проверка полноты цепочки — depth=0 это leaf, depth=1 intermediate, depth=2 root
openssl s_client -connect example.com:443 -servername example.com 2>/dev/null \
  | grep -E "^(depth|verify)"

# Быстрый чек: истекает ли cert менее чем через 30 дней (2592000 секунд)
echo | openssl s_client -connect example.com:443 \
  -servername example.com 2>/dev/null \
  | openssl x509 -noout -checkend 2592000 \
  && echo "cert OK" || echo "WARN: expires in < 30 days"

# Прочитать cert из файла: все поля включая SAN
openssl x509 -in /etc/letsencrypt/live/example.com/cert.pem \
  -noout -text | grep -A4 "Subject Alternative"

# Проверить цепочку локально против CA-bundle
openssl verify -CAfile /etc/ssl/certs/ca-certificates.crt \
  -untrusted intermediate.pem cert.pem
```

**Сгенерировать приватный ключ и CSR для получения сертификата**

```bash
# Генерируем ключ P-256 (короче RSA, не слабее) и CSR.
# CSR отдаётся в CA (или certbot использует его автоматически).
openssl req -newkey ec \
  -pkeyopt ec_paramgen_curve:P-256 \
  -keyout /etc/ssl/private/example.com.key \
  -out /etc/ssl/csr/example.com.csr \
  -nodes \
  -subj "/C=RU/O=My Company/CN=example.com" \
  -addext "subjectAltName=DNS:example.com,DNS:www.example.com"

# Права на ключ: только root читает, nginx запускается от root при старте
chmod 600 /etc/ssl/private/example.com.key

# Самоподписанный cert для локальной разработки (не для прода)
openssl req -x509 -newkey ec \
  -pkeyopt ec_paramgen_curve:P-256 \
  -keyout dev.key -out dev.crt \
  -days 365 -nodes \
  -subj "/CN=localhost" \
  -addext "subjectAltName=DNS:localhost,IP:127.0.0.1"
```

**nginx 1.27 — TLS с Let's Encrypt, OCSP stapling, HSTS**

```nginx
# /etc/nginx/sites-available/example.com

server {
    listen 80;
    server_name example.com www.example.com;

    # HTTP-01 challenge для certbot — путь должен быть доступен до редиректа
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    location / {
        return 301 https://$host$request_uri;
    }
}

server {
    listen 443 ssl;
    http2 on;
    server_name example.com www.example.com;

    # fullchain.pem = leaf + intermediate; без intermediate Android сломается
    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Для OCSP stapling: nginx проксирует статус отзыва вместо браузера
    ssl_trusted_certificate /etc/letsencrypt/live/example.com/chain.pem;
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 1.1.1.1 8.8.8.8 valid=300s;

    # TLS 1.3 предпочтительно, 1.2 для совместимости (старый Safari, Java клиенты)
    ssl_protocols TLSv1.2 TLSv1.3;

    # Только AEAD cipher suites — RC4, 3DES, CBC без GCM выкинуты
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;
    ssl_prefer_server_ciphers off;   # TLS 1.3 сам выбирает оптимальный cipher

    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;         # отключить: ticket key долгосрочный — нет forward secrecy

    # HSTS: браузер 1 год будет ходить только по HTTPS
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;

    location / {
        proxy_pass http://app:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

**mTLS в nginx и в Kubernetes Ingress**

```nginx
# nginx: требовать клиентский сертификат от определённого CA
server {
    listen 443 ssl;
    server_name api-internal.example.com;

    ssl_certificate     /etc/ssl/server.crt;
    ssl_certificate_key /etc/ssl/server.key;

    # CA, который подписывал клиентские сертификаты
    ssl_client_certificate /etc/ssl/client-ca.crt;
    ssl_verify_client on;          # on = обязателен, optional = только проверяет если есть

    # Передаём DN клиентского cert в заголовке — бэкенд может авторизовать по нему
    proxy_set_header X-SSL-Client-DN $ssl_client_s_dn;
    proxy_set_header X-SSL-Client-Verify $ssl_client_verify;

    location / {
        proxy_pass http://backend:8080;
    }
}
```

```yaml
# Kubernetes: nginx-ingress аннотации для mTLS
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-internal
  namespace: production
  annotations:
    # секрет с CA для верификации клиентских сертификатов
    nginx.ingress.kubernetes.io/auth-tls-secret: "production/client-ca-secret"
    nginx.ingress.kubernetes.io/auth-tls-verify-client: "on"
    nginx.ingress.kubernetes.io/auth-tls-verify-depth: "2"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - api-internal.example.com
    secretName: api-internal-tls
  rules:
  - host: api-internal.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: backend
            port:
              number: 8080
```

## Грабли

<ul className="pitfalls">
<li>**Неполная цепочка в `ssl_certificate`** — используешь `cert.pem` вместо `fullchain.pem`. В браузере на десктопе работает: он кешировал intermediate. Android и curl без системного CA-bundle падает с `CERTIFICATE_VERIFY_FAILED`. Фикс: `fullchain.pem` = leaf + intermediate вместе.</li>
<li>**Нет SAN, только CN** — Chrome с 2017 года игнорирует `Common Name` и проверяет только `subjectAltName`. Сертификат с `CN=example.com` без `SAN=DNS:example.com` даёт `ERR_CERT_COMMON_NAME_INVALID`. CSR и self-signed всегда делай с `-addext subjectAltName=...`.</li>
<li>**Wildcard `*.example.com` не покрывает два уровня** — `api.example.com` покрыт, `v2.api.example.com` — нет. Wildcard работает только для одного уровня поддомена. Для глубокой вложенности нужен SAN с явным перечислением.</li>
<li>**Права на ключ `644` вместо `600`** — nginx запустится (читает root-процессом при старте), но security audit зафиксирует проблему, а при смене пользователя запуска nginx сервер не стартанёт. Ключ должен читать только root: `chmod 600 privkey.pem`.</li>
<li>**`ssl_session_tickets on` (дефолт в nginx)** — ticket key хранится в памяти и не ротируется автоматически. Если этот ключ утечёт, расшифровываются все прошлые сессии. Либо отключи (`off`), либо настрой ротацию через `ssl_session_ticket_key` с cron-ротацией.</li>
<li>**Expired intermediate** — Let's Encrypt в сентябре 2021 истёк корневой cert DST Root CA X3. Старые клиенты (OpenSSL < 1.1.0 на Ubuntu 16) перестали принимать сертификаты LE. Урок: следи за цепочкой, не только за leaf. Проверяй `openssl verify` после обновления certbot.</li>
<li>**HTTP-01 challenge с закрытым портом 80** — certbot не может пройти валидацию если 80 заблокирован firewall или сервер за NAT. Переходи на `DNS-01` через `certbot --preferred-challenges dns` и плагин для твоего DNS-провайдера (certbot-dns-cloudflare и т.д.).</li>
<li>**Самоподписанный сертификат в проде** — браузеры показывают «небезопасно», mobile-приложения падают с ошибкой без внятного сообщения, curl отказывается без `--insecure`. Let's Encrypt бесплатный — нет причин держать самоподписанный на публичных эндпоинтах.</li>
</ul>

## Когда что выбирать

| Хочу X | Используй |
| --- | --- |
| Публичный домен, автообновление, бесплатно | `certbot` + Let's Encrypt, `HTTP-01` challenge |
| Сервер за NAT или wildcard-сертификат | Let's Encrypt `DNS-01` challenge |
| Внутренний CA для service-to-service | собственный Intermediate CA (cfssl, step-ca) |
| Аутентифицировать клиента (сервис-к-сервису) | `mTLS` (`ssl_verify_client on` в nginx) |
| Service mesh, zero-trust внутри кластера | Istio / Linkerd — mTLS между всеми подами автоматически |
| TLS 1.3 vs TLS 1.2 | TLS 1.3 предпочтительно; TLS 1.2 для Java 8, старого Safari |
| Один IP, несколько доменов | SNI — поддерживается всеми клиентами с 2010+ |
| Проверить cipher suite на соответствие PCI DSS | `nmap --script ssl-enum-ciphers -p 443 host` |

## Шпаргалка

<div className="cheatsheet">

```bash
# Посмотреть cert сервера: срок, subject, issuer
openssl s_client -connect host:443 -servername host -showcerts 2>/dev/null \
  | openssl x509 -noout -dates -subject -issuer

# Проверить полноту цепочки (depth=0 leaf, depth=1 intermediate)
openssl s_client -connect host:443 -servername host 2>/dev/null \
  | grep -E "^(depth|verify)"

# Истекает ли cert менее чем через 30 дней?
echo | openssl s_client -connect host:443 -servername host 2>/dev/null \
  | openssl x509 -noout -checkend 2592000

# Прочитать cert из файла: все поля, SAN, срок
openssl x509 -in cert.pem -noout -text | grep -A4 "Subject Alt"

# Проверить цепочку локально
openssl verify -CAfile /etc/ssl/certs/ca-certificates.crt \
  -untrusted intermediate.pem leaf.pem

# Сгенерировать ключ P-256 + CSR с SAN
openssl req -newkey ec -pkeyopt ec_paramgen_curve:P-256 \
  -keyout server.key -out server.csr -nodes \
  -subj "/CN=example.com" \
  -addext "subjectAltName=DNS:example.com,DNS:www.example.com"

# Самоподписанный cert для локальной разработки
openssl req -x509 -newkey ec -pkeyopt ec_paramgen_curve:P-256 \
  -keyout dev.key -out dev.crt -days 365 -nodes \
  -subj "/CN=localhost" \
  -addext "subjectAltName=DNS:localhost,IP:127.0.0.1"

# Проверить curl с явным resolve (без DNS, для staging)
curl -v --resolve example.com:443:1.2.3.4 https://example.com/

# Certbot: получить cert (HTTP-01, certbot 2.x)
certbot certonly --webroot -w /var/www/certbot -d example.com -d www.example.com

# Certbot: получить wildcard cert (DNS-01, нужен плагин для DNS)
```
