## Зачем это вообще

Postgres на первый взгляд — «просто реляционная база». На практике один неудачный `SELECT COUNT(*) FROM big_table` без WHERE кладёт прод на 30 секунд, отсутствие нужного индекса делает запрос в 500 раз медленнее, а отключённый autovacuum превращает таблицу в монстра который занимает в 10 раз больше места чем должен. Знать внутреннее устройство не для собеседования — для того чтобы не ждать звонка в 3 ночи.

## Аналогия

<AnalogyBox>
Представь библиотеку, где хранят <b>все черновики каждой книги</b>. Когда читатель берёт книгу, он получает ту версию, которая существовала в момент его прихода — даже если редактор уже правит следующую. Отредактированная версия появляется рядом как новая страница, а старая помечается «устарело». <b>Раз в несколько часов приходит уборщица (vacuum)</b> и выкидывает страницы с пометкой. Если уборщица не справляется — полки переполняются, и чтобы найти нужную книгу, приходится перебирать кучу мусора.
</AnalogyBox>

## Картина мира

<MermaidDiagram label="путь SQL-запроса от клиента до диска" code={`
flowchart LR
  C([клиент]) --> P[Parser<br/>синтаксис]
  P --> R[Rewriter<br/>VIEW, rules]
  R --> PL[Planner<br/>читает pg_stat_*]
  PL --> E[Executor]
  E --> BP[(shared_buffers<br/>buffer pool)]
  BP --> D[(disk<br/>heap + indexes)]
  E --> WAL[(WAL<br/>write-ahead log)]
  WAL --> D
  WAL --> RS[Replica<br/>streaming]
`} />

Planner — самая умная часть: читает статистику из `pg_stat_*` и строит план с минимальной оценочной стоимостью. Устаревшая статистика (не запускали `ANALYZE`) — главная причина плохих планов. WAL пишется раньше heap: именно поэтому база не теряет данные при краше и может реплицироваться.

## Ключевые концепции

<ConceptGrid>
  <ConceptCard title="MVCC + tuple visibility" icon="Copy">
    Каждая строка (tuple) хранит `xmin` и `xmax` — ID транзакций, создавшей и «удалившей» её. Читающая транзакция видит только строки где `xmin <= snapshot_xid`. READ не блокирует WRITE и наоборот. Долгая транзакция держит старый snapshot — vacuum не может убрать мёртвые tuple, пока эта транзакция открыта.
  </ConceptCard>

  <ConceptCard title="WAL (Write-Ahead Log)" icon="FileText">
    Любое изменение сначала пишется в WAL (append-only лог), только потом применяется к heap. Это основа надёжности (crash recovery), point-in-time restore и streaming replication. `wal_level = replica` — минимум для репликации; `logical` нужен для logical replication или дебезиума.
  </ConceptCard>

  <ConceptCard title="Индексы" icon="Search">
    B-tree — дефолт, покрывает `=`, `<`, `>`, `BETWEEN`, `LIKE 'prefix%'`. GIN — для `jsonb`, `tsvector`, массивов: ищет по содержимому. BRIN — для огромных таблиц с естественным порядком (временные ряды), занимает килобайты вместо гигабайт. Hash — только `=`, быстрее B-tree для точечного поиска, но не используется для сортировки. Partial index с `WHERE` сужает размер.
  </ConceptCard>

  <ConceptCard title="EXPLAIN ANALYZE" icon="Lightbulb">
    `EXPLAIN` показывает план без выполнения; `EXPLAIN (ANALYZE, BUFFERS)` — выполняет и показывает реальное время + чтения из shared_buffers / диска. Смотри разницу `rows=5000` (оценка) vs `actual rows=1` — статистика устарела. `Seq Scan` на большой таблице — первый сигнал искать нужный индекс или переписывать запрос.
  </ConceptCard>

  <ConceptCard title="VACUUM и bloat" icon="RefreshCw">
    Обычный VACUUM освобождает место мёртвых tuple для повторного использования внутри таблицы (не возвращает ОС). `VACUUM FREEZE` обновляет `xmin` до спец-значения — защита от transaction ID wraparound (раз в ~2 млрд транзакций). `VACUUM FULL` переписывает таблицу и возвращает место ОС, но берёт `AccessExclusiveLock` — на проде лучше `pg_repack`.
  </ConceptCard>

  <ConceptCard title="Репликация" icon="Shuffle">
    Streaming replication: primary шлёт WAL-поток реплике, та воспроизводит его. Replica в hot standby принимает READ-запросы. Replication slot гарантирует что WAL не удалится до доставки — если replica отстаёт, WAL растёт и диск переполняется. Logical replication реплицирует по таблицам, можно выбирать подмножество; используется для нулевого downtime-миграции версии Postgres.
  </ConceptCard>

  <ConceptCard title="Партиционирование" icon="Layers">
    `PARTITION BY RANGE (created_at)` / `LIST (region)` / `HASH (user_id)`. Planner делает partition pruning: `WHERE created_at > '2025-01-01'` читает только нужные дочерние таблицы. Главная практическая польза — `DROP TABLE partition_old` мгновенная, DELETE с миллионами строк — нет. Индексы, constraints и autovacuum настраиваются на каждой партиции отдельно.
  </ConceptCard>

  <ConceptCard title="Locks" icon="Lock">
    Table-level locks: `ACCESS SHARE` (SELECT) → `ACCESS EXCLUSIVE` (DROP, TRUNCATE, большинство ALTER). `ALTER TABLE ADD COLUMN NOT NULL DEFAULT expr` брало `ACCESS EXCLUSIVE` до PG11 и блокировало таблицу. Row-level: `SELECT ... FOR UPDATE` блокирует строки. Advisory locks — приложение само управляет, удобно для распределённых cron-задач (`pg_try_advisory_lock`).
  </ConceptCard>
</ConceptGrid>

## Жизненный цикл строки при UPDATE

<MermaidDiagram label="что происходит со строкой при UPDATE и VACUUM" code={`
flowchart TD
  A["Original row<br/>xmin=100, xmax=0<br/>(живая)"] --> B

  B["UPDATE: нельзя менять строку на месте<br/>xmax старой = 200<br/>новая строка xmin=200, xmax=0"] --> C

  C{"COMMIT txn 200?"}
  C -- да --> D["Старая строка: xmax=200 (мёртвая)<br/>Новая строка: видна всем > snapshot 200"]
  C -- rollback --> E["Старая строка: xmax сброшен (живая)<br/>Новая строка: удалена"]

  D --> F["Autovacuum: n_dead_tup растёт<br/>видит старую строку — помечает как dead"]
  F --> G["VACUUM: освобождает место<br/>FSM (Free Space Map) обновлён"]
  G --> H([следующий INSERT переиспользует слот])
`} />

UPDATE в Postgres — это DELETE старой строки + INSERT новой. Поэтому UPDATE-heavy таблицы без autovacuum быстро распухают. `n_dead_tup` в `pg_stat_user_tables` — главный показатель.

## Примеры

**Индексы: CONCURRENTLY, partial, составной**

```sql
-- Обычный CREATE INDEX блокирует запись на всё время построения
-- CONCURRENTLY строит в фоне: два прохода, без блокировки
-- Минус: нельзя внутри транзакции, чуть дольше
CREATE INDEX CONCURRENTLY idx_orders_user_created
    ON orders (user_id, created_at DESC);

-- Partial index: только «живые» записи
-- Размер в 10–50x меньше, если deleted_at IS NULL у 5% строк
CREATE INDEX CONCURRENTLY idx_users_active
    ON users (email)
    WHERE deleted_at IS NULL;

-- GIN для поиска по jsonb-полю
CREATE INDEX CONCURRENTLY idx_events_payload
    ON events USING GIN (payload jsonb_path_ops);

-- Проверить что индекс начал использоваться (idx_scan > 0)
SELECT indexrelname, idx_scan
FROM pg_stat_user_indexes
WHERE relname = 'orders'
ORDER BY idx_scan DESC;
```

**EXPLAIN (ANALYZE, BUFFERS) — как читать план**

```sql
-- Всегда с BUFFERS: видно сколько блоков из кеша, сколько с диска
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT u.id, count(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.created_at >= now() - interval '30 days'
GROUP BY u.id
ORDER BY order_count DESC
LIMIT 50;

-- Что смотреть:
--   Seq Scan on orders (cost=0..184000 rows=3200000)
--     → полный скан 3.2M строк — нужен индекс или статистика устарела
--
--   Rows Removed by Filter: 3195000
--     → индекс есть, но почти всё отфильтровывается — partial index
--
--   Buffers: shared hit=480 read=12300
--     → 12300 блоков с диска — shared_buffers мал или данные холодные
--
--   actual rows=1 rows=4800 (estimated)
--     → статистика сильно устарела: ANALYZE users; ANALYZE orders;

-- Обновить статистику без блокировки
ANALYZE users;
ANALYZE orders;
```

**Партиционирование по диапазону дат**

```sql
-- Базовая секционированная таблица
CREATE TABLE events (
    id         bigint       GENERATED ALWAYS AS IDENTITY,
    user_id    bigint       NOT NULL,
    type       text         NOT NULL,
    payload    jsonb,
    created_at timestamptz  NOT NULL DEFAULT now()
) PARTITION BY RANGE (created_at);

-- Партиции создаём заранее (или через pg_partman для автоматизации)
CREATE TABLE events_2025_01 PARTITION OF events
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

CREATE TABLE events_2025_02 PARTITION OF events
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');

-- Индекс на каждой партиции — создаётся отдельно
CREATE INDEX CONCURRENTLY idx_events_2025_01_user
    ON events_2025_01 (user_id, created_at DESC);

-- Проверить что partition pruning работает:
-- в плане должно быть "Partitions: events_2025_01", а не все партиции
EXPLAIN SELECT * FROM events
WHERE created_at >= '2025-01-15' AND created_at < '2025-02-01';

-- Удалить старые данные мгновенно — без блокировки и без bloat
DROP TABLE events_2024_06;
```

**pg_stat_* — мониторинг без сторонних инструментов**

```sql
-- Текущие активные запросы + кто кого блокирует
SELECT pid,
       now() - query_start           AS duration,
       wait_event_type,
       wait_event,
       left(query, 120)              AS query
FROM pg_stat_activity
WHERE state != 'idle'
  AND query_start < now() - interval '3 seconds'
ORDER BY duration DESC;

-- Bloat: таблицы с большим количеством мёртвых строк
SELECT relname,
       n_live_tup,
       n_dead_tup,
       round(n_dead_tup::numeric / nullif(n_live_tup + n_dead_tup, 0) * 100, 1) AS dead_pct,
       last_autovacuum::date,
       last_autoanalyze::date
FROM pg_stat_user_tables
WHERE schemaname = 'public'
ORDER BY n_dead_tup DESC
LIMIT 20;

-- Неиспользуемые индексы — кандидаты на удаление (idx_scan = 0)
SELECT schemaname,
       relname,
       indexrelname,
       idx_scan,
       pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND schemaname = 'public'
ORDER BY pg_relation_size(indexrelid) DESC;

-- Блокировки: кто кого держит
SELECT blocked.pid      AS blocked_pid,
       blocked.query    AS blocked_query,
       blocker.pid      AS blocker_pid,
       blocker.query    AS blocker_query
FROM pg_stat_activity AS blocked
JOIN pg_locks         AS bl  ON bl.pid = blocked.pid AND NOT bl.granted
JOIN pg_locks         AS kl  ON kl.transactionid = bl.transactionid AND kl.granted
JOIN pg_stat_activity AS blocker ON blocker.pid = kl.pid;
```

**Streaming replication: primary + replica**

```ini
# postgresql.conf на primary
wal_level = replica          # минимум для streaming; logical — для logical replication
max_wal_senders = 5          # сколько реплик может подключиться
wal_keep_size = 1GB          # страховка если replica временно отстала
hot_standby = on             # replica принимает READ-запросы
```

```bash
# На replica: создать базовую копию с primary
pg_basebackup \
  --host=primary-host \
  --port=5432 \
  --username=replicator \
  --pgdata=/var/lib/postgresql/16/main \
  --wal-method=stream \
  --checkpoint=fast \
  --progress

# Сигнал: «я replica, воспроизводить WAL»
touch /var/lib/postgresql/16/main/standby.signal

# primary_conninfo в postgresql.conf на replica
# primary_conninfo = 'host=primary-host port=5432 user=replicator'

# Проверить лаг репликации (на primary)
SELECT client_addr,
       state,
       pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS replay_lag
FROM pg_stat_replication;
```

## Грабли

<ul className="pitfalls">
<li>**Индекс на булевой колонке** — B-tree по `is_active boolean` почти бесполезен: только два значения, selectivity ~50%, планировщик выбирает Seq Scan. Используй partial index: `CREATE INDEX ON users (id) WHERE is_active = true`.</li>
<li>**`SELECT COUNT(*) FROM big_table` без WHERE** — полный Seq Scan, блокирует shared_buffers, на таблице 50M строк занимает десятки секунд. Для приблизительного счёта: `SELECT reltuples FROM pg_class WHERE relname = 'big_table'`.</li>
<li>**`ALTER TABLE ADD COLUMN NOT NULL DEFAULT expr`** — до PG11 переписывало всю таблицу с `AccessExclusiveLock`. С PG11+ безопасно, если default — константа (литерал). Нестатичный DEFAULT (например `now()`) всё ещё блокирует. На проде добавляй с `lock_timeout = '2s'`.</li>
<li>**Порядок колонок в составном индексе** — `(user_id, created_at)` покрывает запросы по `user_id` и по `(user_id, created_at)`, но НЕ по одному `created_at`. Planner идёт слева направо: без первой колонки B-tree бесполезен.</li>
<li>**Отключённый autovacuum + горячая таблица** — таблица разбухает: старые tuple не чистятся, size растёт, каждый Seq Scan всё медленнее. `n_dead_tup > 10%` от живых строк при включённом autovacuum — сигнал пересмотреть `autovacuum_vacuum_scale_factor` для этой таблицы.</li>
<li>**`pg_stat_statements` не установлен** — нет реальной аналитики медленных запросов, только `pg_stat_activity` для текущих. Добавь в `shared_preload_libraries = 'pg_stat_statements'` и перезапусти; `CREATE EXTENSION pg_stat_statements` — это должно быть сделано с первого дня.</li>
<li>**Длинная транзакция мешает VACUUM** — открытая транзакция держит старый `xmin`; VACUUM не может убрать dead tuple моложе этого xmin. `SELECT pid, xact_start, state FROM pg_stat_activity WHERE state != 'idle' ORDER BY xact_start` — периодически смотри и завершай транзакции старше 30 минут.</li>
<li>**Миграции без `lock_timeout` и `statement_timeout`** — `ALTER TABLE` встаёт в очередь за долгим SELECT, сам блокирует все следующие запросы. Всегда: `SET lock_timeout = '3s'; SET statement_timeout = '30s';` перед DDL на проде.</li>
</ul>

## Когда что выбирать

| Хочу X | Используй |
| --- | --- |
| Точечный поиск, диапазоны, сортировка | B-tree (дефолт) |
| Поиск по `jsonb`, массиву, `tsvector` | `GIN` |
| Огромная таблица с монотонным ID или timestamp (IoT, логи) | `BRIN` — размер индекса в тысячи раз меньше |
| Только `=` без сортировки, очень высокая кардинальность | `Hash` (редко нужен) |
| Часть строк с фиксированным условием (`deleted_at IS NULL`) | Partial index |
| Реплика для READ-запросов, failover | Streaming replication |
| Реплика на другую версию Postgres или частичная | Logical replication |
| Удаление старых данных без bloat | Partitioning (`DROP TABLE partition`) |
| Шардирование по нагрузке на запись | Citus или application-level sharding |
| Убрать bloat без даунтайма | `pg_repack` (не `VACUUM FULL`) |

## Шпаргалка

<div className="cheatsheet">

```bash
# psql — быстрый осмотр таблицы
\d+ orders               # структура + индексы + размер
\di+ orders              # только индексы таблицы

# Размер таблицы и индексов
SELECT pg_size_pretty(pg_relation_size('orders'));
SELECT pg_size_pretty(pg_total_relation_size('orders'));  # с индексами и toast

# Медленные запросы за всё время (нужен pg_stat_statements)
SELECT left(query, 100), calls, round(mean_exec_time::numeric, 1) AS ms_avg
FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;

# Текущая активность + ожидания
SELECT pid, wait_event_type, wait_event, left(query, 80)
FROM pg_stat_activity WHERE state != 'idle';

# Блокировки
SELECT pid FROM pg_locks WHERE NOT granted;
SELECT pg_cancel_backend(pid);    -- мягко отменить запрос
SELECT pg_terminate_backend(pid); -- завершить сессию

# Bloat
SELECT relname, n_dead_tup, last_autovacuum::date
FROM pg_stat_user_tables ORDER BY n_dead_tup DESC LIMIT 10;

# Дамп / восстановление
pg_dump -Fc -d mydb -f mydb.dump          # custom format, параллельный restore
pg_restore -d mydb -j 4 mydb.dump         # 4 параллельных воркера

# Пересобрать индекс без блокировки записи
REINDEX INDEX CONCURRENTLY idx_orders_user_created;

# Статистика + вакуум вручную
ANALYZE orders;
VACUUM ANALYZE orders;
```
