## Зачем это вообще

Ты можешь настроить один сервер руками за час. Пятьдесят серверов — за неделю, с гарантией что они будут незаметно отличаться друг от друга. Ansible решает это: описываешь желаемое состояние один раз, запускаешь на всех — везде одинаково. Docker закрывает другую проблему: «работает у меня» исчезает, когда приложение вместе с окружением упаковано в образ и запускается на любой машине идентично.

## Аналогия

<AnalogyBox>
Ansible — это шеф-повар, который приходит на общую кухню с рецептом. Он сначала пробует блюдо и добавляет соль только если её не хватает. Запусти рецепт хоть десять раз — блюдо будет одинаковым, пересолить невозможно. <b>Docker</b> — это уже не рецепт, а запечатанный TV-dinner в коробке: внутри уже лежит и сама плита, и кастрюля, и все ингредиенты. Открыл коробку на любом столе — кухня готова к работе.
</AnalogyBox>

## Картина мира

<MermaidDiagram label="Ansible control node → managed hosts / Docker daemon → registry" code={`

  CN -- SSH --> H3[managed host 3]

  subgraph DO[Docker]
    DF([Dockerfile])
    DF --> DB[docker build]

    REG --> DC[docker run / compose]
    DC --> C1[container 1]

Ansible управляет хостами через SSH — агент на хосте не нужен. Docker работает локально через daemon: образ собирается, пушится в registry, оттуда тянется на любой хост и запускается как контейнер.

## Ключевые концепции

<ConceptGrid>
  <ConceptCard title="Идемпотентность" icon="RefreshCw">
    Playbook запускается повторно — уже настроенное не трогается, только дельта применяется.
    Нарушение идемпотентности — первый признак плохого playbook. Если используешь
    `ansible.builtin.shell`, оборачивай в `creates:` или `changed_when: false`, иначе
    модуль будет сообщать `changed` при каждом запуске.
  </ConceptCard>

  <ConceptCard title="Inventory" icon="ListTree">
    Список хостов и групп: `[web]`, `[db]`, `[all]`. Статический — ini или yaml файл.
    Динамический — плагин, который тянет список из AWS/Hetzner/GCP в runtime.
    Переменные хоста — в `host_vars/<hostname>.yml`, группы — в `group_vars/<group>.yml`.
    Никогда не храни пароли в inventory plaintext — только через `ansible-vault`.
  </ConceptCard>

  <ConceptCard title="Playbook и Roles" icon="Workflow">
    Playbook — YAML со списком plays, каждый play = хосты + задачи. Role — переиспользуемый
    блок: `tasks/`, `handlers/`, `defaults/`, `templates/` в одной папке. Если роль
    нужна больше чем в одном playbook — выноси в `roles/`. Иначе достаточно inline tasks.
  </ConceptCard>

  <ConceptCard title="Handlers и notify" icon="Zap">
    Handler — задача, которая запускается только если что-то изменилось. Один handler
    может получить `notify` от десяти задач, но выполнится один раз в конце play.
    Типичный сценарий: изменился конфиг nginx → handler делает `service: state: reloaded`.
    Без handler пришлось бы перезапускать сервис при каждом запуске playbook.
  </ConceptCard>

  <ConceptCard title="Image vs Container" icon="Package">
    Image — неизменяемый слепок файловой системы, шаблон. Container — запущенный процесс
    поверх image с тонким writable-слоем. Один image — тысячи контейнеров одновременно.
    Удалил контейнер — writable-слой исчез, image остался нетронутым.
  </ConceptCard>

  <ConceptCard title="Layers и кеш сборки" icon="Layers">
    Каждая инструкция Dockerfile — отдельный слой. Docker кеширует слои сверху вниз
    и пересобирает начиная с первого изменённого. `COPY . .` перед `npm install` —
    антипаттерн: правка одного `.ts` файла сбрасывает кеш установки зависимостей.
    Правило: тяжёлое и редко меняющееся — ниже, часто меняющееся — выше.
  </ConceptCard>

  <ConceptCard title="Volumes" icon="HardDrive">
    Данные в writable-слое контейнера исчезают при `docker rm`. Volume живёт отдельно
    от контейнера. Bind mount (`./data:/data`) — удобен в dev, но в проде нет гарантии
    прав и структуры директорий. Named volume (`pgdata:/var/lib/postgresql/data`) —
    правильный выбор: Docker сам управляет жизненным циклом.
  </ConceptCard>

  <ConceptCard title="Networks" icon="Network">
    По умолчанию `docker compose` создаёт bridge-сеть и все сервисы видят друг друга
    по имени сервиса. `host` network — контейнер использует сетевой стек хоста, нет
    изоляции, зато нет оверхеда NAT. `none` — полная изоляция. Явно называй сети
    в compose, не полагайся на имя по умолчанию.
  </ConceptCard>
</ConceptGrid>

## Порядок слоёв Dockerfile

<MermaidDiagram label="слои Dockerfile — снизу тяжёлые, сверху лёгкие" code={`

  L3["RUN npm ci --omit=dev

  (исходники — меняются часто)"]

  L6["CMD node dist/index.js
  (метаданные, не создаёт слой с данными)"]

  L1 --> L2 --> L3 --> L4 --> L5 --> L6

  style L1 fill:#1e3a2f,color:#a3e6c3
  style L2 fill:#1e3a2f,color:#a3e6c3

Зелёные слои — стабильны, кешируются на CI. Оранжевые — инвалидируются при каждом изменении исходников. Именно поэтому `COPY package.json` всегда идёт до `COPY .`.

## Примеры

**playbook.yml — установка nginx с handler'ом перезагрузки**

```yaml
---
- name: Настройка веб-сервера
  hosts: web
  become: true           # sudo на удалённом хосте
  become_method: sudo    # явно — не полагаться на дефолт

  vars:
    nginx_version: "1.27.*"

  tasks:
    - name: Установить nginx
      ansible.builtin.apt:
        name: "nginx={{ nginx_version }}"
        state: present
        update_cache: true
        cache_valid_time: 3600   # не делать apt update если кеш свежее часа

    - name: Скопировать конфиг сайта из шаблона
      ansible.builtin.template:
        src: templates/site.conf.j2
        dest: /etc/nginx/sites-available/default
        mode: "0644"
        validate: nginx -t -c %s  # проверить конфиг до замены файла
      notify: Перезагрузить nginx  # handler запустится только если файл изменился

    - name: Убедиться что nginx запущен и включён в автозагрузку
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: true

  handlers:
    - name: Перезагрузить nginx
      ansible.builtin.service:
        name: nginx
        state: reloaded   # reloaded, не restarted — без разрыва соединений
```

**inventory.ini — группы, переменные, bastian-хост**

```ini
[web]
web-01.example.com
web-02.example.com

[db]
db-01.example.com ansible_user=ubuntu

[all:vars]
# python3 обязателен на Ubuntu 22.04+, иначе Ansible использует python2
ansible_python_interpreter=/usr/bin/python3

[web:vars]
# переменные группы; секреты — только через ansible-vault, не здесь
http_port=80
https_port=443
```

**Dockerfile — multi-stage Node-приложение**

```dockerfile
# Stage 1: только зависимости — этот слой кешируется пока не изменится package.json
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev

# Stage 2: сборка
FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
# Исходники копируем ПОСЛЕ node_modules — изменение кода не сбрасывает кеш deps
COPY . .
RUN npm run build

# Stage 3: минимальный финальный образ — только то что нужно в проде
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
# Не запускаем от root — минимальные привилегии
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
COPY --from=deps    /app/node_modules ./node_modules
COPY --from=builder /app/dist         ./dist
COPY package.json ./
USER appuser
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

**docker-compose.yml — web + postgres с healthcheck**

```yaml
services:
  web:
    build: .
    image: myapp:1.0.0    # конкретный тег, не latest
    restart: unless-stopped
    ports:
      - "127.0.0.1:3000:3000"   # bind на localhost — nginx-proxy снаружи, не сам порт
    environment:
      DATABASE_URL: postgres://app:${DB_PASSWORD}@db:5432/appdb
      # Секрет берём из .env файла или окружения хоста — не хардкодим
    depends_on:
      db:
        condition: service_healthy  # ждёт healthcheck, а не просто старт процесса
    networks:
      - backend

  db:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB: appdb
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data   # named volume — данные переживают пересоздание
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d appdb"]
      interval: 5s
      timeout: 3s
      retries: 5
      start_period: 10s   # не считать провалы пока БД не поднялась
    networks:
      - backend

volumes:
  pgdata:

networks:
  backend:
    driver: bridge
```

## Грабли

<ul className="pitfalls">
<li>**`:latest` тег в Dockerfile и compose** — `docker compose up -d --build` не перетянет новый образ если локально уже есть слой с тем же тегом. Тэгай конкретно: `postgres:16-alpine`, `node:20-alpine`. В CI всегда `docker pull` перед билдом или используй `--no-cache`.</li>
<li>**`RUN apt-get update && apt-get install` в отдельных слоях** — если `apt-get update` закешировался, а потом ты добавил пакет, установка упадёт с `Unable to locate package`. Всегда в одной инструкции: `RUN apt-get update && apt-get install -y --no-install-recommends pkg && rm -rf /var/lib/apt/lists/*`.</li>
<li>**`COPY . .` перед `COPY package.json`** — любое изменение `.ts` файла сбрасывает кеш `npm install`. На проекте с 500 зависимостями это плюс 3 минуты к каждому билду. Сначала манифест зависимостей, потом установка, потом исходники.</li>
<li>**`become: true` без `become_method: sudo`** — на некоторых дистрибутивах дефолт не `sudo`, а `su`. Явно прописывай `become_method: sudo` на уровне play или в `ansible.cfg`.</li>
<li>**Забытый `ansible_python_interpreter` на Ubuntu 22.04+** — Ansible по умолчанию ищет `/usr/bin/python`, которого нет. Результат: `Failed to connect to the host via python`. Добавь в `[all:vars]`: `ansible_python_interpreter=/usr/bin/python3`.</li>
<li>**`host_key_checking = True` в CI** — первое подключение к новому хосту падает с интерактивным вопросом про fingerprint, которому никто не ответит. В `ansible.cfg` для CI: `host_key_checking = False`. Для прода — `known_hosts` в CI пайплайне.</li>
<li>**Контейнер работает от `root`** — по умолчанию Docker запускает процесс как root внутри контейнера. Если есть path traversal или RCE — атакующий получает root в контейнере и риск побега. Добавляй `USER appuser` в Dockerfile.</li>
<li>**Секреты через `ENV` в Dockerfile** — `ENV DB_PASSWORD=secret` видно через `docker history myimage`. Передавай секреты через `--secret` флаг buildkit или через переменные окружения хоста в compose, а не запекай в образ.</li>
</ul>

## Когда что выбирать

| Хочу X | Используй |
| --- | --- |
| Настроить и поддерживать конфигурацию хостов | `Ansible` playbook + roles |
| Создать неизменяемое окружение для приложения | `Dockerfile` + image registry |
| Запустить несколько сервисов локально или на одном хосте | `docker compose` |
| Оркестрировать контейнеры на кластере серверов | `Kubernetes` (Ansible тут — только установка K8s) |
| Создать инфраструктуру (VPC, диски, балансировщик) | `Terraform`, не Ansible |
| Установить Terraform, Docker, K8s на хосты | `Ansible` |
| Хранить секреты — пароли, ключи | `ansible-vault` (Ansible) / `--secret` buildkit / Vault (HashiCorp) |
| Дебажить контейнер на лету | `docker exec -it CONTAINER sh` |

## Шпаргалка

<div className="cheatsheet">

```bash
# --- Ansible ---

# проверить связь со всеми хостами из inventory
ansible -i inventory.ini all -m ping

# запустить playbook с конкретным inventory
ansible-playbook -i inventory.ini playbook.yml

# dry-run — показать что изменится, не применять
ansible-playbook -i inventory.ini playbook.yml --check

# запустить только на конкретном хосте или группе
ansible-playbook -i inventory.ini playbook.yml --limit web-01.example.com

# зашифровать файл с секретами
ansible-vault encrypt group_vars/all/secrets.yml

# редактировать зашифрованный файл
ansible-vault edit group_vars/all/secrets.yml

# запустить playbook с vault-паролем из файла
ansible-playbook -i inventory.ini playbook.yml --vault-password-file .vault_pass

# --- Docker ---

# собрать образ с тегом
docker build -t myapp:1.0.0 .

# пересобрать без кеша
docker build --no-cache -t myapp:1.0.0 .

# посмотреть слои образа (и что в каждом)
docker history myapp:1.0.0

# запустить compose в фоне с пересборкой
docker compose up -d --build

# посмотреть логи сервиса в реальном времени
docker logs -f CONTAINER_NAME

# зайти внутрь работающего контейнера
docker exec -it CONTAINER_NAME sh

# удалить остановленные контейнеры, неиспользуемые образы и тома
docker system prune -a --volumes
```

</div>
