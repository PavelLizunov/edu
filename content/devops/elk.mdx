## Зачем это вообще

`grep` и `tail -f` перестают работать когда логи пишут 50 серверов и 100 GB в сутки: ты физически не можешь открыть ssh на каждую машину, а файлы ротируются раньше, чем ты найдёшь нужный момент. ELK Stack собирает логи централизованно, разбирает строки на структурированные поля, хранит с retention-политикой и даёт полнотекстовый поиск за миллисекунды — по индексу, а не сканированием файлов.

## Аналогия

<AnalogyBox>
Elasticsearch — как картотека в библиотеке: каждый лог разбирается на отдельные слова, и для каждого слова создаётся карточка «вот в каких документах оно встречается». Когда ищешь по слову — идёшь в картотеку, а не листаешь все книги подряд. <b>Шард</b> — это отдельный шкаф со своей картотекой, их можно поставить в разные комнаты (ноды). <b>Filebeat</b> — курьер на каждом этаже: забирает конверты (строки логов) и несёт в <b>Logstash</b> — секретаря, который вскрывает каждый конверт и раскладывает по нужным полям перед тем как положить в шкаф.
</AnalogyBox>

## Картина мира

<MermaidDiagram label="app logs → Filebeat → Logstash → Elasticsearch → Kibana" code={`
flowchart LR
  APP([app<br/>/var/log]) -->|tail файлов| FB[Filebeat<br/>лёгкий shipper<br/>на каждом хосте]
  FB -->|Beats protocol<br/>5044| LS[Logstash<br/>grok + geoip<br/>mutate + drop]
  LS -->|HTTP bulk API<br/>9200| ES[(Elasticsearch<br/>cluster<br/>шарды + replicas)]
  ES -->|поиск + агрегации| KB[Kibana<br/>Discover<br/>Dashboards<br/>Alerts]
  ES -.->|ILM<br/>hot→warm→cold→delete| ES
`} />

Logstash — единственный «тяжёлый» компонент: он трансформирует, обогащает и фильтрует. Filebeat намеренно лёгкий (< 50 MB RAM) — ставится на каждый сервер-источник. Elasticsearch держит данные в шардах, распределённых по нодам кластера.

## Ключевые концепции

<ConceptGrid>
  <ConceptCard title="Beats / Filebeat" icon="Plug">
    Семейство лёгких shippers: Filebeat (файлы), Metricbeat (метрики), Packetbeat (сеть).
    Filebeat читает файл как `tail -F`, запоминает позицию в registry, при рестарте продолжает
    с той же строки. Потребляет ~30–50 MB RAM — можно ставить везде без страха.
  </ConceptCard>

  <ConceptCard title="Logstash pipeline" icon="Workflow">
    Три секции: `input` (источник) → `filter` (трансформация) → `output` (куда писать).
    Тяжёлый: запускается на JVM, требует 512 MB+ RAM. Оправдан когда нужен grok, geoip, mutate,
    throttle, aggregate. Если логи уже JSON — можно писать Filebeat прямо в ES, минуя Logstash.
  </ConceptCard>

  <ConceptCard title="Index и Shard" icon="Database">
    Index — логическое хранилище документов, например `nginx-2026.05.20`. Внутри делится на
    primary-шарды (данные) и replica-шарды (копии на других нодах). Шард — минимальная единица
    параллелизма: 1 шард на 1 ноде, разбить нельзя. Слишком много мелких шардов убивает
    cluster state — держи &lt; 30 шардов на ноду.
  </ConceptCard>

  <ConceptCard title="Mapping" icon="ListTree">
    Схема типов полей: `keyword` (точное совпадение, агрегации), `text` (полнотекстовый поиск
    с токенизацией), `ip`, `date`, `integer`. Mapping immutable — создал `message: text`,
    а нужен `keyword`? Придётся reindex. Dynamic mapping удобен при разработке, на проде
    фиксируй через index template.
  </ConceptCard>

  <ConceptCard title="Grok pattern" icon="Search">
    DSL для парсинга строк регулярками: `%{IP:client} %{WORD:method} %{NUMBER:status:int}`
    превращает строку nginx access-лога в структурированные поля. Сотни встроенных паттернов
    (HTTPDATE, URIPATHPARAM, COMMONAPACHELOG). Дебажь в Kibana → Dev Tools → Grok Debugger
    — не трать время на ручной перебор регулярок.
  </ConceptCard>

  <ConceptCard title="ILM — жизненный цикл индекса" icon="Archive">
    Index Lifecycle Management переводит индекс по фазам: hot (запись + чтение, rollover по
    размеру/времени) → warm (только чтение, shrink + forcemerge) → cold (дешёвое хранилище) →
    delete. Без ILM один индекс растёт вечно — reindex становится проблемой на терабайтах.
  </ConceptCard>

  <ConceptCard title="KQL — поиск в Kibana" icon="Filter">
    Kibana Query Language: `status:5* AND method:POST AND duration>1000`. Проще Lucene-синтаксиса.
    `NOT _exists_:error_code` — документы без поля. Wildcards: `host:web-*`. Range: `bytes:[1000 TO *]`.
    В Lens и Dashboard KQL работает как фильтр — можно темплейтить по переменным.
  </ConceptCard>

  <ConceptCard title="Security и roles" icon="ShieldCheck">
    С версии 8.x security включён по умолчанию (TLS между нодами, аутентификация).
    Не отключай через `xpack.security.enabled: false` — это открывает кластер наружу без пароля.
    Роли: `kibana_admin`, `viewer`, `logstash_writer` — минимальные привилегии.
    Пароль для built-in пользователей задаётся через `elasticsearch-setup-passwords auto`.
  </ConceptCard>
</ConceptGrid>

`} />

## Примеры

**filebeat.yml — читаем nginx-логи, мультилайн для java стектрейсов, шлём в Logstash**

```yaml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/nginx/*.log
    tags: ["nginx"]   # по тегу Logstash поймёт какой grok применять

  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
    tags: ["java-app"]
    # Java стектрейсы: строка НЕ начинается с даты → склеиваем с предыдущей
    multiline.pattern: '^\d{4}-\d{2}-\d{2}'
    multiline.negate: true
    multiline.match: after
    multiline.max_lines: 100   # защита от бесконечного склеивания

output.logstash:
  hosts: ["logstash:5044"]
  # TLS к Logstash (рекомендован в проде)
  ssl.certificate_authorities: ["/etc/filebeat/ca.crt"]

# Не шли внутренние метрики Filebeat в тот же кластер — они шумят
monitoring.enabled: false
```

**logstash/pipeline/nginx.conf — grok + date + geoip → Elasticsearch**

```text
input {
  beats {
    port => 5044
    ssl  => true
    ssl_certificate => "/etc/logstash/logstash.crt"
    ssl_key         => "/etc/logstash/logstash.key"
  }
}

filter {
  if "nginx" in [tags] {
    grok {
      match => {
        "message" => '%{IPORHOST:client_ip} - %{DATA:user} \[%{HTTPDATE:log_timestamp}\] "%{WORD:method} %{URIPATHPARAM:request} HTTP/%{NUMBER:http_version}" %{NUMBER:status:int} %{NUMBER:bytes:int} "%{DATA:referrer}" "%{DATA:user_agent}"'
      }
      # Не теряем документ при фейле — помечаем и всё равно пишем
      tag_on_failure => ["_grok_failure"]
    }

    date {
      match => ["log_timestamp", "dd/MMM/yyyy:HH:mm:ss Z"]
      target => "@timestamp"
      remove_field => ["log_timestamp"]   # убираем дубль
    }

    geoip {
      source => "client_ip"
      target => "geoip"
      # fields — только то что нужно, не тащи всё
      fields => ["city_name", "country_code2", "location"]
    }

    # Удаляем сырое сообщение после успешного парсинга — экономим место
    if "_grok_failure" not in [tags] {
      mutate { remove_field => ["message"] }
    }
  }
}

output {
  elasticsearch {
    hosts    => ["https://elasticsearch:9200"]
    index    => "nginx-%{+yyyy.MM.dd}"
    user     => "logstash_writer"
    password => "${LOGSTASH_ES_PASSWORD}"
    ssl_certificate_verification => true
    cacert   => "/etc/logstash/ca.crt"
  }
}
```

**ILM-политика + index template с явным mapping**

```json
PUT _ilm/policy/nginx-logs-policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "50gb",
            "max_age":  "1d"
          }
        }
      },
      "warm": {
        "min_age": "3d",
        "actions": {
          "shrink":     { "number_of_shards": 1 },
          "forcemerge": { "max_num_segments": 1 }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": { "delete": {} }
      }
    }
  }
}

PUT _index_template/nginx-logs-template
{
  "index_patterns": ["nginx-*"],
  "template": {
    "settings": {
      "number_of_shards":   3,
      "number_of_replicas": 1,
      "index.lifecycle.name":             "nginx-logs-policy",
      "index.lifecycle.rollover_alias":   "nginx"
    },
    "mappings": {
      "dynamic": "strict",
      "properties": {
        "@timestamp": { "type": "date" },
        "client_ip":  { "type": "ip" },
        "method":     { "type": "keyword" },
        "request":    { "type": "keyword" },
        "status":     { "type": "short" },
        "bytes":      { "type": "integer" },
        "user_agent": { "type": "text", "fields": { "keyword": { "type": "keyword", "ignore_above": 512 } } },
        "referrer":   { "type": "keyword" },
        "geoip": {
          "properties": {
            "city_name":    { "type": "keyword" },
            "country_code2":{ "type": "keyword" },
            "location":     { "type": "geo_point" }
          }
        }
      }
    }
  }
}
```

**KQL-запросы в Kibana Discover**

```text
# Все 5xx от POST-запросов медленнее 1 секунды
status:5* AND method:POST AND duration>1000

# Конкретный IP за последние 30 минут (time filter отдельно в UI)
client_ip:"203.0.113.42"

# Документы где поле error_code вообще существует
_exists_:error_code

# Wildcard по хосту
host.name:web-*

# Диапазон байт
bytes:[102400 TO *]

# Поиск по geo — страна
geoip.country_code2:CN AND status:4*
```

**curl против ES API — диагностика кластера**

```bash
# Состояние кластера (green / yellow / red)
curl -s -u elastic:${ES_PASSWORD} \
  https://localhost:9200/_cat/health?v

# Индексы, отсортированные по размеру
curl -s -u elastic:${ES_PASSWORD} \
  "https://localhost:9200/_cat/indices?v&s=store.size:desc&h=index,health,pri,rep,docs.count,store.size"

# Почему шарды unassigned — главный инструмент диагностики
curl -s -u elastic:${ES_PASSWORD} \
  https://localhost:9200/_cluster/allocation/explain | python3 -m json.tool

# Поиск с body (последние 10 ошибок)
curl -s -u elastic:${ES_PASSWORD} \
  -H "Content-Type: application/json" \
  https://localhost:9200/nginx-*/_search -d '{
    "query": { "term": { "status": 500 } },
    "sort":  [{ "@timestamp": "desc" }],
    "size":  10
  }' | python3 -m json.tool

# Reindex из старого индекса в новый (например после изменения mapping)
curl -s -u elastic:${ES_PASSWORD} \
  -X POST -H "Content-Type: application/json" \
  https://localhost:9200/_reindex -d '{
    "source": { "index": "nginx-old" },
    "dest":   { "index": "nginx-new" }
  }'
```

## Грабли

<ul className="pitfalls">
<li>**Dynamic mapping взрывает схему** — поле `user_id` пришло числом, потом той же строкой пришла строка `"anon"` → ES отвергает документ с ошибкой mapping conflict. На проде ставь `"dynamic": "strict"` в template и явно описывай все поля.</li>
<li>**Слишком много мелких шардов** — один шард потребляет ~30 MB кучи на JVM. Кластер из 3 нод с 1000 индексами по 5 шардов = 5000 шардов = ~150 GB только под управление. Правило: &lt; 30 шардов на 1 GB heap ноды. Используй rollover + ILM вместо ежедневных ручных индексов.</li>
<li>**Mapping immutable после создания** — создал `message: text`, а нужен `keyword` для term-запросов? `term` вернёт пустой результат потому что text-поле токенизируется. Нужна связка `text` + `keyword` subfield или полный reindex.</li>
<li>**ILM rollover alias не создан** — Logstash пишет в `nginx-2026.05.20` напрямую, а не через alias `nginx`. ILM не видит индекс через alias и rollover никогда не срабатывает. Один индекс растёт бесконечно.</li>
<li>**Стектрейсы без multiline в Filebeat** — каждая строка Java exception = отдельный документ в ES. Grok не парсит фрагменты, в Kibana 50 документов вместо одного события. Решение — `multiline` конфиг в Filebeat до отправки в Logstash.</li>
<li>**Диск 95% → кластер блокирует запись** — ES включает flood-stage watermark при заполнении диска и переводит индекс в read-only. Логи перестают записываться, Logstash начинает буферить. Настрой алерт на 80%, а не жди 95%. Разблокировка: `PUT index/_settings {"index.blocks.read_only_allow_delete": null}`.</li>
<li>**`_source: false` в mapping** — экономит место, но без `_source` нельзя сделать reindex или update. Если понадобится изменить mapping — придётся читать данные из другого источника. Не отключай `_source` если не уверен что данные никогда не понадобятся заново.</li>
<li>**Security отключён в проде** — ES 7.x по умолчанию открыт без аутентификации. В 8.x security включён по умолчанию, но миграция с 7.x часто ломает это. Не добавляй `xpack.security.enabled: false` в конфиг «чтобы быстрее заработало» — это открывает кластер без пароля.</li>
</ul>

## Когда что выбирать

| Хочу X | Используй |
| --- | --- |
| Логи с нескольких серверов, нужен grok/enrichment | Filebeat → Logstash → ES |
| Логи уже в JSON, enrichment не нужен | Filebeat → ES напрямую (без Logstash) |
| Метрики (CPU, диск) вместе с логами | Metricbeat + Filebeat в один ES |
| Логи в Kubernetes, нет shell-доступа к нодам | Fluent Bit (DaemonSet) → ES или Loki |
| Prometheus уже есть, хочу логи рядом | Grafana Loki (дешевле хранилище, слабее поиск) |
| Облако, не хочу управлять кластером | Elastic Cloud или OpenSearch Serverless |
| Долгосрочное дешёвое хранение (холодный архив) | ILM cold phase + Searchable Snapshots в S3 |
| Поиск по `keyword` полю точным совпадением | `term` запрос |
| Полнотекстовый поиск по `text` полю | `match` запрос |

```bash
# Состояние кластера
curl -s -u elastic:$ES_PASS https://localhost:9200/_cat/health?v

# Создать / обновить ILM-политику
curl -s -u elastic:$ES_PASS -X PUT -H "Content-Type: application/json" \

# Создать index template
curl -s -u elastic:$ES_PASS -X PUT -H "Content-Type: application/json" \
  https://localhost:9200/_index_template/nginx-logs-template -d @template.json

curl -s -u elastic:$ES_PASS -X POST -H "Content-Type: application/json" \
  https://localhost:9200/_reindex \

# Разблокировать read-only индекс (после flood-stage)
curl -s -u elastic:$ES_PASS -X PUT -H "Content-Type: application/json" \

# Проверить конфиг Filebeat без запуска
filebeat test config -c /etc/filebeat/filebeat.yml

# Тест конфига Logstash
/usr/share/logstash/bin/logstash -t -f /etc/logstash/conf.d/nginx.conf

# Mapping индекса
curl -s -u elastic:$ES_PASS https://localhost:9200/nginx-2026.05.20/_mapping | python3 -m json.tool
```
