## Зачем это вообще

Реляционная база хранит данные нормализованно: один объект — десятки таблиц, JOIN на JOIN. Когда данные естественно собираются в агрегаты — заказ с позициями, профиль пользователя с историей — документная модель читает их за один запрос без сборки по частям. MongoDB оправдан там, где схема меняется быстро и объекты нечасто пересекаются. «Schemaless» — это маркетинг: схема у тебя всё равно есть, только лежит в коде клиента, а не в DDL базы. Это означает, что несовместимые изменения формата тихо ломают старые документы.

## Аналогия

<AnalogyBox>
Представь папку с бланками, где каждый бланк посвящён одному клиенту: имя, адрес, список заказов — всё вместе. Найти всё о клиенте — вытащил один бланк. Обычная папка (реляционная база) хранит имена в одном ящике, адреса в другом, заказы в третьем — приходится бегать и склеивать. Минус бланков: <b>если один телефон записан у двух клиентов, поменял у одного — у второго устарел</b>. Ещё есть хитрость: папку можно разрезать на стопки по первой букве фамилии и разложить по разным шкафам — это <b>sharding</b>. Но если ты плохо выбрал, по какому признаку резать, все новые клиенты будут сыпаться в один шкаф, а остальные пустуют.
</AnalogyBox>

Запись идёт в primary того шарда, который определяется по shard key. Чтение — с любой реплики при `readPreference: secondaryPreferred`. Config servers хранят маппинг «диапазон shard key → шард»; mongos кеширует его локально.

## Ключевые концепции

<ConceptGrid>
  <ConceptCard title="Document & Collection" icon="FileText">
    Документ — BSON-объект до 16 МБ. Embedding (вложенный объект) работает когда данные всегда читаются вместе. Reference (`ObjectId`) нормализует, но требует второго запроса или `$lookup`. Правило: embed когда данные «принадлежат» родителю, reference когда живут самостоятельно и часто меняются отдельно.
  </ConceptCard>

  <ConceptCard title="Индексы" icon="Search">
    Без индекса — COLLSCAN по всей коллекции. Compound `{a:1, b:1}` работает по prefix: покрывает запросы по `a` и `(a, b)`, но не по одному `b`. Multikey строится автоматически для массивов. Text-индекс для полнотекстового поиска. Hashed — только для равенства, не для range. Partial-индекс уменьшает размер, индексируя только нужное подмножество документов.
  </ConceptCard>

  <ConceptCard title="Aggregation Pipeline" icon="Filter">
    Цепочка этапов: `$match` → `$group` → `$sort` → `$limit` → `$lookup` → `$project`. `$match` в начале использует индекс; `$match` в конце — full scan остатка. `$lookup` — аналог LEFT JOIN, дорогой, особенно без индекса на `foreignField`. `$facet` позволяет параллельно считать несколько агрегаций в одном пайплайне.
  </ConceptCard>

  <ConceptCard title="Replica Set" icon="RefreshCw">
    Primary принимает записи, secondaries реплицируются асинхронно. При падении primary — election (~10–30 с). `writeConcern: { w: "majority" }` гарантирует, что запись подтверждена большинством. Без `wtimeoutMS` приложение зависнет навсегда если шард недоступен. Чтение с secondary через `readPreference` даёт stale data — учитывай для кеш-сценариев.
  </ConceptCard>

  <ConceptCard title="Sharding и shard key" icon="Shuffle">
    Shard key — immutable, выбирается один раз на всю жизнь коллекции. Хороший ключ: высокая кардинальность + равномерное распределение + не монотонно возрастающий. ObjectId как shard key — плохо: все вставки бьют в один шард (hotspot). Hashed shard key распределяет равномерно, но делает range-запросы scatter-gather по всем шардам.
  </ConceptCard>

  <ConceptCard title="Read / Write Concerns" icon="ShieldCheck">
    `writeConcern: { w: 1 }` — подтверждение только от primary (быстро, риск потери при failover). `{ w: "majority" }` — надёжно, но медленнее. `readConcern: "majority"` — читаешь только закоммиченные большинством данные, защита от грязного чтения. `"local"` — быстрее, но может вернуть данные которые потом откатились.
  </ConceptCard>

  <ConceptCard title="Transactions" icon="Shield">
    Multi-document ACID-транзакции с MongoDB 4.0. Дорогие: локи, WAL, rollback, таймауты. Правильная стратегия — проектировать так, чтобы атомарная операция укладывалась в один документ. Транзакция — запасной вариант, не базовый. В sharded cluster транзакции ещё дороже: координация через mongos и config servers.
  </ConceptCard>

  <ConceptCard title="Explain & Диагностика" icon="Lightbulb">
    `explain("executionStats")` показывает: `stage` (IXSCAN хорошо, COLLSCAN плохо), `executionTimeMillis`, `totalDocsExamined` vs `totalDocsReturned`. Если examined >> returned — запрос гоняет лишние документы, нужен индекс или уточнение фильтра. `db.currentOp()` — живые запросы, `db.serverStatus()` — метрики инстанса.
  </ConceptCard>
</ConceptGrid>

## Жизненный цикл / пайплайн

<MermaidDiagram label="aggregation pipeline: как работает $match + индекс" code={`
flowchart LR
  C[(collection<br/>1M docs)] --> M
  M["$match<br/>{ status: 'active',<br/>  createdAt: ≥ 2026-01-01 }"]
  M -- "IXSCAN по индексу<br/>{status,createdAt}" --> G
  note1([если $match первый<br/>— используется индекс]) -.-> M
  G["$group<br/>{ _id: '$userId',<br/>  total: $sum }"]
  G --> S["$sort<br/>{ total: -1 }"]
  S --> L["$limit 10"]
  L --> LK["$lookup<br/>from: 'users'"]
  LK --> P["$project<br/>убрать лишние поля"]
  P --> R([результат<br/>10 строк])
  note2([если $match в конце<br/>— full scan 1M docs]) -.-> G
`} />

`$match` до `$group` режет выборку до агрегации — это принципиально. Поменяй их местами — и вместо 500 документов через индекс агрегатор прогонит миллион.

## Примеры

**Индексы: unique, compound, partial, explain**

```javascript
// Уникальный: дубли по email блокируются на уровне БД, не в приложении
db.users.createIndex({ email: 1 }, { unique: true })

// Compound: покрывает (tenant), (tenant + createdAt), но не одиночный createdAt
// -1 = убывающий порядок — для ORDER BY createdAt DESC
db.events.createIndex({ tenant: 1, createdAt: -1 })

// Partial: индексируем только активные документы
// Архивные (status != 'active') не занимают место в индексе
db.orders.createIndex(
  { userId: 1, createdAt: -1 },
  { partialFilterExpression: { status: { $in: ["pending", "processing", "active"] } } }
)

// Проверить использование индекса — ищем IXSCAN, не COLLSCAN
// totalDocsExamined должно быть близко к totalDocsReturned
db.orders.find({ userId: "u123", status: "pending" }).explain("executionStats")
```

**Aggregation pipeline с $lookup, $unwind, $group, $facet**

```javascript
db.orders.aggregate([
  // Всегда первым — срезает объём до всех дальнейших этапов
  {
    $match: {
      createdAt: { $gte: new Date("2026-04-01"), $lt: new Date("2026-05-01") },
      status: "completed"
    }
  },
  // JOIN с коллекцией users — индекс на users._id обязателен
  {
    $lookup: {
      from:         "users",
      localField:   "userId",
      foreignField: "_id",
      as:           "user"
    }
  },
  // $lookup возвращает массив — $unwind делает из него один объект
  { $unwind: "$user" },
  // Параллельно считаем два агрегата в одном проходе
  {
    $facet: {
      topBuyers: [
        { $group: { _id: "$userId", name: { $first: "$user.name" }, total: { $sum: "$amount" } } },
        { $sort: { total: -1 } },
        { $limit: 10 }
      ],
      byCategory: [
        { $group: { _id: "$category", revenue: { $sum: "$amount" }, count: { $sum: 1 } } },
        { $sort: { revenue: -1 } }
      ]
    }
  }
])
```

**Explain: читать вывод executionStats**

```javascript
// Запускаем explain и смотрим на ключевые поля

  { status: "active", region: "eu" }

// inputStage.stage == "SORT"            — in-memory sort, нет индекса по sort-полю

// Если examined >> returned — нужен индекс или уточнение $match
// Если stage == "SORT" — добавь sort-поле в конец compound index
```

**Replica set: инициализация и управление**

```javascript
// Инициализировать replica set из mongosh на первом узле
rs.initiate({
  _id: "rs0",
  members: [
    { _id: 0, host: "mongo1:27017", priority: 2 },  // preferred primary
    { _id: 1, host: "mongo2:27017", priority: 1 },
    { _id: 2, host: "mongo3:27017", arbiterOnly: true }  // голосует, не хранит данные
  ]
})

// Состояние: кто primary, кто secondary, replication lag
rs.status()

// Подробнее: отставание каждого secondary в секундах
rs.printSecondaryReplicationInfo()

// Принудительный stepdown primary (на 60 с) — для обслуживания без даунтайма
rs.stepDown(60)

// Добавить новый узел в существующий replica set
rs.add("mongo4:27017")
```

**Sharding: включение, shard key, мониторинг**

```javascript
// Включить sharding для базы данных
sh.enableSharding("myapp")

// Создать hashed shard key — равномерно распределяет вставки
// Но делает range-запросы scatter-gather по всем шардам
sh.shardCollection("myapp.events", { userId: "hashed" })

// Range shard key — хорош для range-запросов, плох для hotspot при вставке
// НЕ используй монотонно возрастающий ключ (ObjectId, timestamp) как range key
sh.shardCollection("myapp.logs", { tenantId: 1, createdAt: 1 })

// Статус шардирования: балансировка, chunks, мигрирующие chunk'и
sh.status()

// Сколько chunk'ов на каждом шарде (должно быть примерно равно)
db.adminCommand({ listShards: 1 })

// Текущие операции — найти долгие запросы
db.currentOp({ "secs_running": { $gt: 5 } })
```

## Грабли

<ul className="pitfalls">
<li>**ObjectId или timestamp как range shard key** — ObjectId монотонно возрастает, поэтому все новые документы идут в один шард (hotspot write). Балансировщик не успевает, один шард перегружен. Решение: hashed shard key или составной ключ с полем высокой кардинальности в начале.</li>
<li>**Нет индекса на shard key** — запрос без shard key превращается в scatter-gather: mongos рассылает запрос на все шарды, собирает и мержит результаты. Медленно и нагружает весь кластер, а не один шард.</li>
<li>**Compound multikey с двумя массивами** — MongoDB запрещает документ где оба поля compound-индекса являются массивами. Декартово произведение индексных записей слишком велико. Вставка вернёт ошибку `cannot index parallel arrays`. Один массив — ок, два — нет.</li>
<li>**`writeConcern: majority` без `wtimeoutMS`** — если шард недоступен, операция зависнет навсегда. Всегда добавляй `{ w: "majority", wtimeoutMS: 5000 }`. Иначе приложение заморозится при failover.</li>
<li>**`$where` с JavaScript-функцией** — выполняется на сервере в JS-движке, без индексов, медленно и небезопасно. Всегда заменяется на нативные операторы MongoDB. С MongoDB 7.0 `$where` официально deprecated.</li>
<li>**`findAndModify` в цикле вместо `bulkWrite`** — каждый вызов `findAndModify` — отдельный round-trip. Для массовых обновлений используй `db.collection.bulkWrite([...])`: один запрос, на порядок быстрее.</li>
<li>**Нет `--auth` в production** — по умолчанию MongoDB слушает без аутентификации. Если порт 27017 торчит наружу или в сеть где не должен — данные читает любой. Всегда запускай с `--auth` и создавай пользователей с минимальными правами.</li>
<li>**Отключённый journaling ради производительности** — journaling защищает от потери данных при крэше (аналог WAL в Postgres). Отключение даёт ~10% прироста, но при неожиданном перезапуске данные последних записей теряются. Не делай так на данных, которые нельзя потерять.</li>
</ul>

## Когда что выбирать

| Хочу X | Используй |
| --- | --- |
| Данные естественно собираются в агрегаты (заказ с позициями, профиль) | Document model, embedding |
| Высоконормализованные данные с множеством связей (финансы, ERP) | PostgreSQL вместо MongoDB |
| Равномерное распределение вставок по шардам | `hashed` shard key |
| Range-запросы по shard key (логи по времени внутри tenant) | Range shard key (`{ tenantId: 1, ts: 1 }`) |
| Нагрузка на чтение, можно немного stale | `readPreference: "secondaryPreferred"` |
| Нагрузка до ~100 ГБ, нет горизонтального роста | Replica set без sharding |
| Нагрузка ТБ+, горизонтальный рост | Sharded cluster |
| Связанные данные в разных коллекциях, редкий JOIN | `$lookup` в aggregation pipeline |
| Атомарность нескольких документов обязательна | Multi-document transaction |
| Полнотекстовый поиск по полю | `text` index + `$text` оператор |
