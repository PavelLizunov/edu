## Зачем это вообще

Когда инфраструктуру создают руками через веб-консоль — это называют «click-ops». Работает до первого аудита, инцидента или просьбы «подними точно такое же в staging». Воспроизвести окружение по памяти невозможно, проследить кто что изменил — тоже. Terraform описывает инфраструктуру кодом: один и тот же конфиг разворачивает prod и staging, все изменения проходят через git, а drift между желаемым и реальным состоянием видно через `plan`.

## Аналогия

<AnalogyBox>
Представь архитектора с подробным чертежом дома. <b>Чертёж</b> — это твой HCL-конфиг, он описывает каждую комнату. <b>Фотография построенного</b> — это state-файл, фиксирует что реально стоит на участке прямо сейчас. Перед каждым выездом архитектор сверяет чертёж с фотографией и составляет список работ: что достроить, что снести, что сдвинуть — это <b>plan</b>. Когда ты говоришь «поехали» — он вызывает бригаду и строители делают ровно этот список — это <b>apply</b>. Если сосед поменял дверь без ведома архитектора, фотография устарела и следующий plan покажет это как расхождение — <b>drift</b>.
</AnalogyBox>

## Картина мира

<MermaidDiagram label="terraform code → state → cloud" code={`
flowchart LR
  subgraph local[локальная машина / CI]
    CFG[HCL конфиг<br/>.tf файлы]
    TF([terraform plan/apply])
    CFG --> TF
  end
  subgraph backend[remote backend]
    S3[(S3<br/>terraform.tfstate)]
    DDB[(DynamoDB<br/>lock table)]
    S3 <--> DDB
  end
  subgraph cloud[cloud provider]
    AWS[AWS / GCP / Azure<br/>реальные ресурсы]
  end
  TF -- читает state --> S3
  TF -- ставит lock --> DDB
  TF -- создаёт/изменяет/удаляет --> AWS
  TF -- пишет новый state --> S3
  TF -- снимает lock --> DDB
`} />

`plan` читает state и опрашивает API облака — сравнивает оба. `apply` фиксирует lock в DynamoDB, применяет изменения в cloud и записывает новый state в S3. Без lock два параллельных `apply` запишут state поверх друг друга и получат повреждённый файл.

## Ключевые концепции

<ConceptGrid>
  <ConceptCard title="Provider" icon="Plug">
    Плагин-драйвер, который знает API конкретного облака или сервиса. `hashicorp/aws`, `hashicorp/google`, `hashicorp/kubernetes`. Без провайдера Terraform не знает как разговаривать с AWS. Версию провайдера всегда фиксируй через `~> 5.0` — мажорные обновления ломают ресурсы без предупреждения.
  </ConceptCard>

  <ConceptCard title="Resource" icon="Server">
    Описание одного управляемого объекта: `aws_s3_bucket`, `aws_instance`, `aws_route53_record`. Terraform сравнивает желаемое состояние (конфиг) с текущим (state + реальный API) и приводит одно к другому. Один `resource` — одна единица управления, один CRUD-цикл.
  </ConceptCard>

  <ConceptCard title="Data source" icon="Database">
    Read-only ссылка на ресурс, которым Terraform не управляет. `data "aws_ami" "ubuntu"` — не создаёт AMI, только читает ID последней Ubuntu. Позволяет ссылаться на существующую инфраструктуру (созданную вручную или другим Terraform-проектом) без передачи её под управление.
  </ConceptCard>

  <ConceptCard title="State" icon="HardDrive">
    Файл `terraform.tfstate` — единственный источник истины о том, что Terraform уже создал и чем управляет. Хранит ID ресурсов в облаке, их атрибуты и зависимости между ними. Без state Terraform не знает, что удалять при `terraform destroy`. Local state опасен в команде — один `apply` поверх другого гарантированно даст конфликт.
  </ConceptCard>

  <ConceptCard title="Plan / Apply" icon="GitBranch">
    `plan` — dry-run: показывает diff между конфигом+state и реальным облаком. `+` создать, `~` изменить, `-` удалить. `apply` исполняет этот diff. Никогда не делай `apply` без просмотра `plan`: удаление RDS-инстанса с данными видно только в плане, и это последний шанс остановиться.
  </ConceptCard>

  <ConceptCard title="Module" icon="Layers">
    Переиспользуемый набор ресурсов — папка с `.tf`-файлами, которую вызываешь как функцию с переменными. Вместо копирования VPC-конфига в три окружения — один модуль `module "vpc"` с параметром `env`. Публичные модули есть в Terraform Registry, но лучше начинать со своих: публичные часто overengineered.
  </ConceptCard>

  <ConceptCard title="Backend + Lock" icon="Lock">
    Backend определяет, где лежит state. S3 + DynamoDB — стандарт для AWS: S3 хранит файл, DynamoDB атомарно блокирует его на время `apply`. Без блокировки параллельный `apply` от коллеги повредит state — придётся восстанавливать вручную через `terraform state` команды.
  </ConceptCard>

  <ConceptCard title="Workspaces" icon="Layers">
    Один набор `.tf`-файлов, несколько изолированных state-файлов. `terraform workspace new staging` создаёт `staging`-state в том же backend. Удобно для лёгких окружений (dev/staging/prod с одним кодом), но при росте инфры лучше разделять по директориям — workspaces не дают изоляции провайдера и конфига.
  </ConceptCard>
</ConceptGrid>

## Жизненный цикл ресурса

<MermaidDiagram label="от изменения кода до state update" code={`
flowchart TD
  C1([изменил .tf конфиг]) --> C2
  C2[terraform plan<br/>читает state + опрашивает cloud API] --> C3
  C3{diff найден?}
  C3 -- нет --> C3n([No changes. Infrastructure is up-to-date.])
  C3 -- да --> C4[показывает diff<br/>+ create / ~ update / - destroy]
  C4 --> C5{ты смотришь plan<br/>и говоришь yes}
  C5 -- отмена --> C5n([изменения не применены])
  C5 -- подтвердил --> C6[terraform apply<br/>lock state в DynamoDB]
  C6 --> C7[вызов cloud API<br/>создать / изменить / удалить ресурс]
  C7 --> C8{cloud ответил ок?}
  C8 -- ошибка --> C8e[apply останавливается<br/>state частично обновлён<br/>следующий plan покажет что сделано]
  C8 -- ок --> C9[записать новый state в S3<br/>снять lock]
  C9 --> C10([готово, ресурс в желаемом состоянии])
`} />

## Примеры

**main.tf — S3-бакет и базовая IAM-политика**

```hcl
terraform {
  required_version = ">= 1.7"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # мажор фиксируем, минор обновляем свободно
    }
  }
}

provider "aws" {
  region = var.aws_region
}

resource "aws_s3_bucket" "app_assets" {
  bucket = "${var.project}-${var.env}-assets"

  # destroy заблокирует удаление через Terraform — защита от случайного destroy
  lifecycle {
    prevent_destroy = true
  }
}

resource "aws_s3_bucket_versioning" "app_assets" {
  bucket = aws_s3_bucket.app_assets.id

  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "app_assets" {
  bucket = aws_s3_bucket.app_assets.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

# Читаем текущий аккаунт без создания ресурса — data source
data "aws_caller_identity" "current" {}

output "bucket_arn" {
  value       = aws_s3_bucket.app_assets.arn
  description = "ARN бакета для IAM политик других модулей"
}
```

**backend.tf — remote state на S3 с DynamoDB lock**

```hcl
# bootstrap: этот файл нужен ПОСЛЕ того как S3 + DynamoDB созданы
# в отдельном Terraform-проекте (или через AWS Console один раз)
terraform {
  backend "s3" {
    bucket         = "my-company-tfstate"
    key            = "prod/app/terraform.tfstate"
    region         = "eu-central-1"
    dynamodb_table = "my-company-tfstate-lock"
    encrypt        = true   # AES256 на стороне S3
  }
}
```

**variables.tf + terraform.tfvars — типизированные переменные**

```hcl
# variables.tf — декларация с типами и валидацией
variable "aws_region" {
  type        = string
  default     = "eu-central-1"
  description = "AWS регион деплоя"
}

variable "env" {
  type        = string
  description = "Окружение: dev | staging | prod"

  # если передать staging123 — terraform plan сразу упадёт с ошибкой
  validation {
    condition     = contains(["dev", "staging", "prod"], var.env)
    error_message = "env должен быть одним из: dev, staging, prod"
  }
}

variable "project" {
  type    = string
  default = "myapp"
}
```

```hcl
# terraform.tfvars — значения для конкретного окружения
# в git не коммитим prod-значения если там секреты
aws_region = "eu-central-1"
env        = "prod"
project    = "myapp"
```

**Использование модуля + outputs**

```hcl
# вызов локального модуля (./modules/vpc)
module "vpc" {
  source = "./modules/vpc"

  env        = var.env
  cidr_block = "10.0.0.0/16"
  az_count   = 3   # сколько availability zones

  # теги наследуются всеми ресурсами внутри модуля
  tags = {
    Project     = var.project
    Environment = var.env
    ManagedBy   = "terraform"
  }
}

# output модуля используем снаружи
resource "aws_instance" "app" {
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t3.small"
  subnet_id     = module.vpc.private_subnet_ids[0]   # из output модуля
}

# terraform output vpc_id — покажет значение после apply
output "vpc_id" {
  value = module.vpc.vpc_id
}
```

## Грабли

<ul className="pitfalls">
<li>**State хранится в git** — `terraform.tfstate` содержит все атрибуты ресурсов в plaintext, включая пароли RDS и ключи IAM. Даже если удалить из HEAD, в истории останется навсегда. Только remote state (S3, Terraform Cloud).</li>
<li>**Ручное удаление ресурса в облаке** — state знает о нём, реальность — нет. Следующий `plan` предложит его пересоздать. Если пересоздание нежелательно — сначала `terraform state rm resource.name`, потом удаляй в облаке.</li>
<li>**`terraform destroy` без `-target`** — уничтожит всё что в state данного проекта, включая базы с данными. На проде делай только `terraform destroy -target=aws_instance.staging_worker`.</li>
<li>**Смешение `count` и `for_each` на одном ресурсе** — при удалении элемента из середины `count`-списка Terraform пересчитывает индексы и пересоздаёт все последующие ресурсы. Используй `for_each` с map/set: удаление одного ключа не трогает остальные.</li>
<li>**Версии провайдеров не запинены** — `version = ">= 5.0"` позволяет поставить 6.0 при следующем `init`, а 6.0 может сломать синтаксис ресурсов. Используй `~> 5.0` (разрешает 5.x, запрещает 6.x) и коммить `.terraform.lock.hcl`.</li>
<li>**Sensitive-значения попадают в state** — `sensitive = true` в переменной скрывает значение в выводе `plan`, но в `terraform.tfstate` оно хранится plaintext. Для паролей БД используй AWS Secrets Manager + data source, а не прямую передачу через переменную.</li>
<li>**Отсутствие lock при параллельном apply** — два инженера запустили `apply` одновременно без DynamoDB-блокировки. Оба прочитали одинаковый state, оба записали разные версии поверх. Итог: state не отражает реальность и нужен `terraform state` для ручного восстановления.</li>
<li>**`terraform import` без написания конфига** — `import` добавляет ресурс в state, но не создаёт `.tf`-описание. Следующий `plan` сразу предложит удалить его (нет в конфиге). Сначала пишешь ресурс в конфиге, потом импортируешь.</li>
</ul>

## Когда что выбирать

| Хочу X | Используй |
| --- | --- |
| Провиженить облачные ресурсы (VPC, RDS, S3, IAM) | `Terraform` |
| Настроить ОС, поставить пакеты на уже существующие серверы | `Ansible` |
| Инфраструктура на TypeScript/Python/Go, не хочу учить HCL | `Pulumi` |
| Инфра только в AWS, нет мульти-клауда, нет внешней команды | `CloudFormation` / `CDK` |
| Переиспользовать конфиг VPC в dev/staging/prod | `module` с переменными |
| Лёгкая изоляция state при одном конфиге | `terraform workspace` |
| Полная изоляция окружений (разные провайдеры, конфиги) | отдельные директории / Terragrunt |
| Посмотреть что изменится перед деплоем в CI | `terraform plan -out=plan.tfplan` + `show` |

## Шпаргалка

<div className="cheatsheet">

```bash
# инициализация: скачивает провайдеры, настраивает backend
terraform init
terraform init -upgrade   # обновить провайдеры до последних в пределах constraint

# diff конфига с реальностью — читать перед каждым apply
terraform plan
terraform plan -out=plan.tfplan           # сохранить plan для apply в CI
terraform apply plan.tfplan               # применить сохранённый plan (без вопросов)

# применить изменения
terraform apply
terraform apply -auto-approve             # только в CI, не на проде руками
terraform apply -target=aws_s3_bucket.x  # только один ресурс

# уничтожить (осторожно)
terraform destroy
terraform destroy -target=aws_instance.staging

# форматирование и валидация
terraform fmt -recursive
terraform validate

# состояние state
terraform state list
terraform state show aws_s3_bucket.tfstate
terraform state rm aws_instance.legacy    # убрать из state без удаления в cloud
terraform state mv module.old module.new  # переименовать без пересоздания

# импорт существующего ресурса в state
terraform import aws_s3_bucket.legacy my-existing-bucket-name

# пометить ресурс как «нужно пересоздать» при следующем apply
terraform taint   aws_instance.web   # устарело в 1.0, замена ниже
terraform apply -replace=aws_instance.web

# workspaces
terraform workspace list
terraform workspace new staging
terraform workspace select prod
```
