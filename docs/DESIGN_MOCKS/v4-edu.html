#Kubernetes

#Kubernetes

От подов до Ingress: оркестрация контейнеров на проде. Декларативно, эфемерно, масштабируемо.

From pods to Ingress: container orchestration in prod. Declarative, ephemeral, scalable.

аналогияanalogy

💡 представь💡 imagine
💡
Представь дирижёра большого оркестра: каждый музыкант — это контейнер, группа струнных — это под, партитура — это Deployment. Дирижёр сам не играет, но следит чтобы нужное число скрипок звучало в нужный момент, и заменяет того кто сбился. Imagine the conductor of a large orchestra: each musician is a container, the strings section is a pod, the score is a Deployment. The conductor doesn't play, but watches that the right number of violins sound at the right moment — and replaces anyone who falls out of step.

ключевые концепцииkey concepts

Pod

Pod

Минимальная единица: один или несколько контейнеров с общей сетью и томами.

Minimal unit: one or more containers sharing network and volumes.

Deployment

Deployment

Декларативное описание желаемого состояния: «всегда 3 пода, такие-то».

Declarative description of desired state: "always 3 pods, like this".

Service

Service

Стабильная точка входа к подам — даже когда сами поды пересоздаются.

Stable entry point to pods — even when pods themselves get recreated.

Ingress

Ingress

HTTP-роутинг снаружи внутрь: host/path → нужный Service.

HTTP routing from outside in: host/path → the right Service.

★ кредыcreds

ConfigMap / Secret

Конфиги и креды отдельно от образа — монтируются как env или файлы.

Configs and credentials separate from the image — mounted as env vars or files.

scale

HPA

Авто-масштабирование подов по CPU/памяти или кастомным метрикам.

Pod auto-scaling by CPU/memory or custom metrics.

примерыexamples

deployment.yamlYAML
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels: { app: web }
  template:
    metadata:
      labels: { app: web }
    spec:
      containers:
        - name: web
          image: ghcr.io/me/web:1.4.2
          ports: [{ containerPort: 8080 }]
          resources:
            requests: { cpu: 100m, memory: 128Mi }
            limits:   { cpu: 500m, memory: 512Mi }
рaскатка & статусrollout & statusBASH
# применить и подождать раскатку
$ kubectl apply -f deployment.yaml
$ kubectl rollout status deployment/web --timeout=2m

# откатить если что-то пошло не так
$ kubectl rollout undo deployment/web

# потыкать поды живьём
$ kubectl get pods -l app=web -o wide
$ kubectl logs -f -l app=web --tail=100
$ kubectl exec -it $(kubectl get pod -l app=web -o name | head -1) -- sh

проверь себяcheck yourself

quiz · 01

Под перезапустился. Почему его IP-адрес теперь другой?

A pod was restarted. Why does it now have a different IP?

→ Pod-уровень IP не гарантируется. Поэтому клиенты ходят через Service (стабильный DNS-имя + ClusterIP), а сам Service следит за актуальным набором подов через label-селектор. Никакой ротации «для безопасности» нет — это просто свойство эфемерности. Pod-level IPs aren't guaranteed. That's why clients go through a Service (stable DNS name + ClusterIP), and the Service tracks the current pod set via a label selector. There's no "rotation for security" — it's just ephemerality.

#Ansible & Docker

Конфигурация и упаковка. Idempotent playbook + immutable image — это два слоя одной идеи: «опиши состояние, не последовательность шагов».

Configuration and packaging. Idempotent playbook + immutable image — two layers of the same idea: "describe state, not a sequence of steps".

аналогияanalogy

💡
💡
Ansible — это рецепт для общей кухни (хост уже стоит, нужно довести до нужного состояния). Docker — это запечатанный TV-dinner: внутри уже всё нужное, разогрел и работает. Ansible is a recipe for a shared kitchen (the host already exists, bring it to a known state). Docker is a shrink-wrapped TV-dinner: everything is inside, heat it up, done.

#Terraform

Инфраструктура как код. State, plan, apply, drift — и почему state-файл нельзя терять.

Infrastructure as code. State, plan, apply, drift — and why you must not lose the state file.

аналогияanalogy

💡
📐
Terraform — это чертёж дома. State — это «фотография уже построенного». Plan показывает разницу. Apply вызывает строителей. Drift — это когда сосед поставил тебе новую дверь без спроса. Terraform is a house blueprint. State is a "photo of what's already built". Plan shows the diff. Apply calls the builders. Drift is the neighbour installing a new door without asking.

#СетиNetworks

OSI, маршрутизация, DNS. Где находится пакет, и зачем там столько слоёв.

OSI, routing, DNS. Where the packet is, and why there are so many layers.

аналогияanalogy

💡
📦
Пакет — это посылка в матрёшке: Ethernet кадр обёртывает IP пакет, тот обёртывает TCP сегмент, тот — твой HTTP запрос. Каждый слой умеет одно и доверяет соседним. A packet is a nested parcel: an Ethernet frame wraps an IP packet, which wraps a TCP segment, which wraps your HTTP request. Each layer does one thing and trusts the neighbours.

#TLS

Рукопожатие, сертификаты, mTLS. Кто кого подписывает и зачем.

Handshake, certificates, mTLS. Who signs whom and why.

аналогияanalogy

💡
🛂
Сертификат — это паспорт. Центр сертификации — это паспортный стол, которому ты доверяешь. Рукопожатие — это пограничный пункт: показал паспорт, договорились про общий язык, дальше говорите шёпотом. A certificate is a passport. The CA is the passport office you trust. The handshake is border control: show the passport, agree on a shared language, then whisper.

edu/devops/postgresql

#PostgreSQL

Индексы, MVCC, репликация. Что такое vacuum и почему он важен.

Indexes, MVCC, replication. What vacuum is and why it matters.

аналогияanalogy

💡
📚
MVCC — это библиотека с черновиками: каждая транзакция читает свой снимок мира. Новая запись = новая страница. Удалить — это пометить «не актуально». Vacuum — это уборщица, которая выносит вычеркнутые страницы. MVCC is a library with drafts: each transaction reads its own snapshot of the world. A new write = a new page. Delete = mark "obsolete". Vacuum is the cleaner who takes out the crossed-out pages.

edu/devops/mongodb

#MongoDB

Схема (несмотря на лозунги), шардинг, индексы.

Schema (despite slogans), sharding, indexes.

аналогияanalogy

💡
📂
«Без схемы» — это маркетинг. Реальность: схема всё равно есть, только она в коде клиента. Mongo гибко принимает документы, но шарду нужен shard key, и плохой ключ — это плохая база до конца жизни. "Schemaless" is marketing. Reality: there is still a schema, just in the client. Mongo flexibly accepts documents, but a shard needs a shard key, and a bad key is a bad database for life.

edu/devops/elk

#ELK

Индексация, шарды, retention. И почему логи дороже, чем кажутся.

Indexing, shards, retention. And why logs are more expensive than they look.

аналогияanalogy

💡
📇
Elasticsearch — это картотека: каждый лог разбирается на слова, каждое слово получает карточку с указанием «вот в каких документах оно встречается». Поиск — пошёл по картотеке, нашёл документы. Шард — отдельный шкаф с картотекой. Elasticsearch is a card catalog: each log is broken into words, each word gets a card listing "the documents I appear in". Search = walk the catalog, find the documents. A shard is one cabinet of cards.
