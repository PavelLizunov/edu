#Kubernetes

От подов до Ingress: оркестрация контейнеров на проде.

аналогия

💡

Представь дирижёра большого оркестра: каждый музыкант — это контейнер, группа струнных — это под, партитура — это Deployment. Дирижёр сам не играет, но следит чтобы нужное число скрипок звучало в нужный момент, и заменяет того кто сбился.

ключевые концепции

Pod

Минимальная единица: один или несколько контейнеров с общей сетью и томами.

Deployment

Декларативное описание желаемого состояния: «всегда 3 пода, такие-то».

Service

Стабильная точка входа к подам — даже когда сами поды пересоздаются.

Ingress

HTTP-роутинг снаружи внутрь: host/path → нужный Service.

ConfigMap / Secret

Конфиги и креды отдельно от образа — монтируются как env или файлы.

HPA

Авто-масштабирование подов по CPU/памяти или кастомным метрикам.

примеры

deployment.yamlyaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels: { app: web }
  template:
    metadata:
      labels: { app: web }
    spec:
      containers:
        - name: web
          image: ghcr.io/me/web:1.4.2
          ports: [{ containerPort: 8080 }]
          resources:
            requests: { cpu: 100m, memory: 128Mi }
            limits:   { cpu: 500m, memory: 512Mi }
rollout & statusbash
# применить и подождать раскатку
$ kubectl apply -f deployment.yaml
$ kubectl rollout status deployment/web --timeout=2m

# откатить если что-то пошло не так
$ kubectl rollout undo deployment/web

проверь себя

Под перезапустился. Почему его IP-адрес теперь другой?

→ Pod-уровень IP не гарантируется. Поэтому клиенты ходят через Service (стабильный DNS-имя + ClusterIP), а сам Service следит за актуальным набором подов через label-селектор.
