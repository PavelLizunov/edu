# Методология автономной мульти-агентной разработки

## Цель
За ~10 часов автономной работы (от старта после получения дизайна) построить рабочий MVP `edu.ninitux.com` с 8 темами DevOps, развёрнутый на проде.

## Состав «команды»

Один оркестратор (главный агент в Claude Code) + специализированные subagent'ы:

| Subagent             | Модель  | Назначение                                       |
|----------------------|---------|--------------------------------------------------|
| `topic-author`       | sonnet  | Пишет один MDX-файл темы с нуля                  |
| `content-reviewer`   | sonnet  | Ревьюит MDX — техника, структура, стиль          |
| `component-builder`  | sonnet  | Пишет один React-компонент или страницу          |
| `ui-reviewer`        | sonnet  | Ревьюит компоненты — a11y, responsive, Tailwind  |
| `Explore`            | (built) | Точечный поиск по кодбазе                        |
| `general-purpose`    | (built) | Сложные многошаговые исследования                |

## Принципы

### 1. Один эталон → копия
Прежде чем штамповать 8 тем, делается **одна эталонная тема** (Kubernetes), которая проходит полное ревью. На её основе фиксируется шаблон. Только потом — параллельно остальные 7.

То же с компонентами: сначала один `ConceptCard` идёт через `ui-reviewer`, исправляется, тогда же остальные 3 (`AnalogyBox`, `CodeBlock`, `QuizCard`) пишутся по этому стилю.

### 2. Параллелизм
Где можно — независимые задачи запускаются как несколько `Agent` tool-calls в одном сообщении. Например, 7 оставшихся тем после эталона — это 7 параллельных `topic-author` вызовов.

### 3. Quality gates
Никакая работа не считается завершённой пока не пройдёт:
- `bun run lint` — чисто
- `bun run type-check` — чисто
- `bun run build` — чисто
- Соответствующий reviewer (content или ui) — ✅

Если gate упал — фикс на месте, не переходить к следующей задаче.

### 4. Append-only прогресс-лог
Каждая итерация дописывает строчку в `docs/PROGRESS_LOG.md`:
```
[ISO timestamp] iter NNN | <category>: <short desc> | gate: ✅/⚠️ | files: a.tsx, b.mdx
```
Лог нужен чтобы пользователь, придя через 10 часов, видел путь работы.

### 5. Коммиты — мелкие
- После каждой завершённой темы — commit
- После каждого компонента — commit
- После каждой настройки конфига — commit

Conventional Commits. Это даёт чёткую историю и возможность откатить точечно.

### 6. Никакого "выглядит готово" без проверки
- Build должен запускаться на CI или хотя бы локально перед "финалом"
- Деплой на прод выполняется только когда оркестратор может ответить "да" на:
  - сборка проходит
  - все 8 тем имеют все 4 секции
  - lint и type-check чистые
  - mobile-вид проверен (resize в браузере / DevTools mobile preview через CDP на 192.168.0.142:9222)

## Анти-паттерны (чего НЕ делать)

1. **Не писать 8 тем подряд без ревью.** Низкое качество расползётся.
2. **Не игнорировать ошибки lint/type.** Подавление ошибок (`@ts-ignore`, `eslint-disable`) запрещено без явной причины в комментарии.
3. **Не добавлять зависимости "на всякий случай".** Каждая зависимость = тяжелее bundle, риск supply-chain.
4. **Не обновлять Next.js / Tailwind / shadcn посреди работы.** Зафиксировал версии — работаешь с ними.
5. **Не деплоить если не локально не зашёл и не пощёлкал.** Headless Chrome (192.168.0.142:9222) — для самопроверки.

## Use Headless Chrome для self-testing

После build:
```bash
# через CDP подключение к 192.168.0.142:9222
# навигация на http://localhost:18900 (proxy через ssh tunnel) или edu.ninitux.com если уже задеплоено
# скриншоты ключевых экранов в /tmp/edu-screenshots/
```

Это даёт оркестратору обратную связь от настоящего рендера, не только статического кода.

## Ритм работы

- 1 итерация ≈ 5–8 минут (один атомарный таск + ревью + коммит)
- ~75–80 итераций за 10 часов с учётом перекуров и параллельных вызовов
- Резерв 20 итераций — на фиксы reviewer'ов и непредвиденное

## Запуск автономного режима

После получения дизайна и сохранения в `docs/DESIGN_TOKENS.md`:

```
/loop <<autonomous-loop-dynamic>>
```

В каждой итерации цикла оркестратор:
1. Читает `docs/ITERATION_PLAN.md` — текущий шаг
2. Делает работу (сам или через subagent)
3. Проверяет gate
4. Пишет в `docs/PROGRESS_LOG.md`
5. Коммитит
6. Переходит к следующему шагу

## Стоп-условия

Цикл останавливается если:
- План выполнен полностью (все шаги ✅)
- Подряд 3 итерации с упавшим gate — оркестратор фиксирует "нужен человек" в логе и останавливается
- Изменения вышли за scope (например, попытка изменить инфру)
